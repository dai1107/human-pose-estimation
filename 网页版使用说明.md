# HYROX 多人摄像头网页版使用说明

网页版允许电脑和手机访客使用各自设备的摄像头进行实时 HYROX 姿态分析，并在每次动作完成后播报需要改进的地方。不同访客使用独立匿名会话，动作设置、语音开关、上传文件和分析结果互不共享。

## 1. 安装与启动

首次使用先安装依赖：

```powershell
.venv\Scripts\python.exe -m pip install -r requirements.txt
```

`requirements.txt` 会安装已验证版本的核心依赖和 YOLO；也可分别安装
`requirements-core.txt` 与 `requirements-yolo.txt`。执行
`python -m pip install -e .` 后，还可以用 `pose-web`、`pose-doctor` 和
`pose-clean` 代替对应的 Python 模块命令。

以上依赖可使用纯 MediaPipe 和 YOLO + MediaPipe。启用 `YOLO + RTMW WholeBody（高精度 133 点）` 时，还需安装 CPU 或 GPU 版本的 ONNX Runtime（二选一）：

```powershell
# CPU：精度不变，但 RTMW-X/L 实时帧率较低
.venv\Scripts\python.exe -m pip install -r requirements-rtmw-cpu.txt

# NVIDIA GPU：实时摄像头和手机访问推荐
.venv\Scripts\python.exe -m pip uninstall -y onnxruntime onnxruntime-gpu
.venv\Scripts\python.exe -m pip install -r requirements-rtmw-gpu.txt
```

不要在同一虚拟环境中同时保留 `onnxruntime` 和 `onnxruntime-gpu`。RTMW 权重约 229 MB，不提交到 Git；文件名、官方地址和 Provider 检查命令见 [模型安装说明](models/README.md)。

### 本机使用

双击 `启动网页.bat`，等待浏览器自动打开；也可以在命令行运行：

```powershell
.venv\Scripts\python.exe start_web.py
```

程序默认打开 `http://127.0.0.1:5000`。该地址只有在启动窗口保持运行时才能访问，关闭启动窗口就会停止服务。

`localhost` 和 `127.0.0.1` 属于浏览器认可的安全上下文，可以在当前电脑使用摄像头。程序显示的局域网 HTTP 地址可用于查看页面或上传视频，但现代浏览器通常不会允许另一台设备在非 HTTPS 地址使用摄像头。

### 临时公网分享

双击 `启动公网访问.bat`，等待窗口输出 `https://...trycloudflare.com/` 地址，再将地址发给需要使用的人。仓库中的 `tools\cloudflared.exe` 是该入口所需程序。

默认是匿名临时访问，不需要登录或邀请码。如需给本次分享添加随机访问口令：

```powershell
.venv\Scripts\python.exe start_public_web.py --protected
```

带口令的完整地址会显示在窗口中并写入快捷方式。不要单独转发去掉查询参数后的地址。

Quick Tunnel 地址每次重启都会变化，只适合测试和临时分享，不是正式长期网址。中国大陆稳定公网服务仍需选定国内可访问的服务器、备案或合规域名、长期 HTTPS 入口和生产 WebSocket 服务。

## 2. 使用本机摄像头

1. 在“画面来源”中选择“本机摄像头”。
2. 阅读页面中的摄像头用途与隐私说明。
3. 点击“开启摄像头”，在浏览器原生弹窗中允许视频权限。
4. 选择摄像头设备；手机可使用“切换前 / 后”。
5. 选择 HYROX 动作、拍摄视角和灵敏度；需要时在“高级设置”中调整识别模型、显示选项或手动地板线。
6. 在“动作反馈”右上角确认“语音开”或按需关闭语音。
7. 点击“开始实时分析”，查看骨架、人体规则计数或分析周期、阶段、反馈和网络状态；每完成一个可播报候选后会即时播报改进建议。
8. 可以暂停分析、截取当前画面或下载分析报告。
9. 点击“停止摄像头”后，网页会立即停止并释放所有视频轨道。

网页未点击“开启摄像头”前不会申请权限，并始终使用 `{ video: ..., audio: false }`，不会请求麦克风。切换设备、停止摄像头、关闭页面或页面离开时会释放原摄像头。

拍摄时建议让一名主要运动者完整入镜、固定设备并保持光照充足。正面视角适合对称性和膝部轨迹，侧面视角适合躯干角度、髋铰链和前后阶段。

### 识别模型选择

高级设置提供三种可手动选择的模型方案，“自动选择”是选择策略，不是第四种模型：

| 页面选项 | 关键点与推荐用途 | 计算位置 |
|---|---|---|
| `纯 MediaPipe` | 33 个姿态点；适合普通单人实时分析 | 服务器执行，通常使用 CPU；不会按动作隐式加载 YOLO |
| `YOLO + MediaPipe` | YOLO 核心身体点加同人 MediaPipe 足部点；适合多人画面中的目标锁定 | 服务器 CPU/GPU；所有动作都固定运行双模型融合 |
| `YOLO + RTMW WholeBody（高精度 133 点）` | 身体、脚部、面部和双手关键点；推荐用于 Lunge、Wall Ball | 服务器推理，推荐 NVIDIA GPU |
| `自动选择（推荐）` | 按动作选择标准后端；Lunge 可使用 YOLO + MediaPipe | 不自动启用 RTMW；需要高精度时请手动选择 |

手机不会下载或直接运行 RTMW 权重，而是把摄像头帧发送给网页服务器推理。服务器性能决定主要推理速度；手机配置仍会影响摄像头质量、视频编码、页面绘制、发热和网络传输。

### 网页怎样判断动作问题

网页不会自动判断用户正在做哪项 HYROX 动作。用户选择动作后，浏览器把无音频的视频帧发送给服务器；服务器使用所选姿态后端识别并平滑人体关键点，再将关键点转换为膝、髋、肘和肩角度、躯干角、身体中心、归一化位置、可见度以及必要的触地和脚步事件。

动作专属状态机先确认当前处于站立、最低点、投掷、起跳、落地等哪个阶段，再执行该阶段适用的规则。实时“中/高”灵敏度允许一个已处理帧确认短暂关键端点，但完整端点顺序、计数冷却和可观测性仍会过滤抖动、遮挡和孤立失败帧。完整判断链路见 [README 的“动作问题如何判断”](README.md#动作问题如何判断)。

页面各区域的含义如下：

| 页面内容 | 数据来源与含义 |
|---|---|
| 骨架颜色 | 当前阶段明显偏离或命中高置信度问题时显示红色，通过时显示绿色，接近边界或证据不足时显示黄色/中性；部分不评价技术质量的过渡阶段也保持绿色 |
| 关节旁角度 | 显示当前动作关心的膝、髋、肘或躯干角度；颜色只反映当前阶段适用标准 |
| 动作反馈 | 动作分析器输出的优先技术问题，默认最多两项；可见度不足时优先提示调整取景 |
| 动作标准 | 解释当前实现使用的官方人体代理和训练参考，不等于现场器械、目标或距离规则 |
| 有效/未完成/无法确认 | 完整候选经必需规则聚合和证据质量门控后的 `VALID/NO_REP/UNSURE` |
| 语音提示 | 汇总刚完成动作段中的清晰角度偏离和技术问题；报告会记录问题出现的画面数，完整计数仍单独执行证据质量门控 |

技术提示和有效计数是两层结果。例如膝内扣、轻微躯干前倾、节奏偏快或小幅不对称通常作为训练反馈保留，不会单独取消计数；后膝触地、Wall Ball 深度/伸展、Burpee 起落和前向位移等动作专属必需规则才参与完整候选判定。如果拍摄视角、地板或关键点不足，网页应显示 `UNSURE`，而不是把看不清的动作当成 `NO_REP`。

### 实时摄像头的计数与分析周期标准

实时摄像头在“中/高”灵敏度下允许一个已处理帧确认快速关键端点，避免浏览器发送或推理队列丢帧时错过投球、起跳、落地等短暂姿态。完整端点顺序、通用接触防抖、脚部事件防抖和计数冷却仍用于防止重复判断；“低”灵敏度更保守，需要连续多帧。

| 动作 | 网页实时判定的关键顺序 | 取景重点与语义 |
|---|---|---|
| 负重箭步蹲 | 站立 → 后膝触地确认 → 触地后完全伸展 | 侧面或斜侧面，全身与脚下地板入镜；侧面使用更可见的同侧腿链判断伸展，并要求触地腿交替且无额外调整步。 |
| Wall Ball | 站直起始 → 髋低于膝 → 向上完全伸展 → 双手投掷代理 | 正面或斜前方；双脚、地板、髋、肩和双腕持续入镜，决定 `VALID/NO_REP/UNSURE`。 |
| Rowing | Catch → Finish | 侧面，完整拍到膝、髋、肩、肘和手腕；只增加分析 `cycle_count`。 |
| SkiErg | 顶部 → 底部 → 顶部 | 正面或斜前方，双手腕全程入镜；只增加分析 `cycle_count`。 |
| Burpee Broad Jump | 胸部触地确认 → 双脚同步起跳 → 双脚同步落地 → 下一次 hands-down 验证 | 侧面约 45°，保留当前和下一落地区域；落地后继续检查补步、手部位置与前向位移。 |
| Sled Push | Drive → Step | 侧面或斜侧面，完整拍到躯干、膝和脚踝；每个推动步只增加分析 `cycle_count`。 |
| Sled Pull | Reach → Pull → Recover → Reach | 侧面或斜侧面，完整拍到双臂、膝和脚下地板；后拉和随后的向前回正合为一个分析周期，回到前伸位后才增加 `cycle_count`。 |
| Farmers Carry | 连续搬运状态 | 不按次数拆分；双手、髋、膝和脚踝都应入镜，查看搬运状态和持续违规提示。 |

手动选择 `YOLO + MediaPipe` 时，所有动作都会使用同一条融合管线：YOLO 负责锁定目标运动员，MediaPipe 最多检测 5 个人体候选，并用两种模型共有的鼻、肩、肘、腕、髋、膝和踝进行身份匹配。只有匹配到 YOLO 目标的 MediaPipe 候选才可补充脚跟、脚尖等专有节点；匹配失败时不会选用背景人物。MediaPipe 短时漏检期间，脚部点按“最近一次可靠脚点相对当前 YOLO 脚踝的偏移”有限保持，并随时间降低置信度，过期后仍按不可观测处理。

如需更可靠地观察脚跟、脚尖和手指，可在“高级设置 → 识别模型”中选择 `YOLO + RTMW WholeBody（高精度 133 点）`。YOLO 先锁定唯一目标运动员，RTMW-X/L 只在该人物框内输出 WholeBody 133 点；系统再用 13 个共有身体节点核验身份，匹配成功后才接受 RTMW 的脚部和双手各 21 点。模型文件或 ONNX Runtime 不可用时，页面会明确显示 `YOLO + MediaPipe（RTMW 降级）`，不会静默使用背景人物或直接中断训练。

Wall Ball 如果预览正常但有效动作为 0，依次检查：动作是否选为 Wall Ball、视角是否为正面/斜前方、开始时双髋双膝是否完全伸展、脚下地板是否可靠、最低点是否明确髋低于膝、投掷端点是否完全伸展，以及双手腕是否均从胸前上升到肩部以上。默认阈值见 [HYROX 动作配置说明](configs/hyrox/README.md#默认中灵敏度计数与分析周期端点)。

“实时结果”以识别到的“完整动作周期”为主计数，并同时分列有效动作、未完成和无法确认。对于人体规则计数动作，四项满足：完整动作周期 = 有效动作 + 未完成 + 无法确认。`NO_REP` 表示证据充分且必需人体规则明确失败；`UNSURE` 表示遮挡、视角、地板、置信度或关键时序不足，不能可靠判定。技术质量提示与必需规则分开处理，轻微姿态偏差不会自动取消计数。

Rowing、SkiErg、Sled Push 和 Sled Pull 的通用结果卡片沿用相同字段，但其中的数字只代表分析周期，必须结合 `count_semantics: analysis_cycle` 和 `official_rep_count_supported: false` 解读，不能视为官方有效次数、完成距离或雪橇过线。Farmers Carry 使用连续监控语义，次数保持 0。

### 距离动作的人体违规代理

- Rowing 仅在用户点击开始到停止分析的区间内检测持续站起代理 `ROWING_EARLY_STAND_PROXY`；正面等不适合的视角不会输出明确违规；
- SkiErg 不识别底座，因此不会用脚离地本身判违规；
- Sled Push 不以某个固定关节角度或躯干姿势判比赛违规；
- Sled Pull 可检测拉动阶段持续跪姿 `SLED_PULL_KNEELING_VIOLATION` 和明确坐姿 `SLED_PULL_SEATED_VIOLATION`；坐姿几何成立但地板或膝接触证据不足时显示 `UNSURE_POSSIBLE_SEATED_PULL`，而不是明确违规；
- Farmers Carry 在搬运移动期间检测手臂未基本伸展 `ARM_NOT_EXTENDED_VIOLATION` 或未位于身体两侧 `ARM_NOT_BY_SIDE_VIOLATION`；
- 所有明确持续违规都经过时间门控，正常步态或单帧关键点抖动不会立即判错。

### 高级显示设置

“高级设置”中的显示选项分为骨架范围和三个同级开关：

- **骨架设置**：选择完整骨架、仅上半身或仅下半身；下拉列表不再包含“隐藏面部”；
- **显示手指节点**：RTMW WholeBody 启用时直接使用其双手 21 点；其他模型使用独立 MediaPipe 手部模型。开启后每只手显示五根手指的 20 个非手腕关节点，默认开启；
- **隐藏面部**：隐藏面部姿态节点，默认关闭；
- **镜像画面**：控制画面左右翻转，摄像头自拍时通常更直观。摄像头模式默认开启，示例和上传视频默认关闭。

### 局部地板线

系统优先在用户站直且双脚稳定时，自动采集最近约 0.5 秒的脚跟和脚尖位置建立局部地板线。负重箭步蹲和 Wall Ball 若没有完整站姿，也可使用连续稳定的支撑脚标定地板，但不会把屈膝姿态当作站立身高。相机画面明显倾斜时，先开启摄像头预览，再点击“高级设置 → 两点标定”，依次点击脚下地板上两个相距较远的点。绿色虚线表示参考可用，橙色表示当前参考不确定。

地板线只用于比较膝、髋、胸部虚拟表面点等人体部位与脚下地面的相对高度，不代表真实厘米测量或接触面积。双脚长期被遮挡、人体未完整入镜、相机移动或标定线与脚部位置矛盾时，系统会返回 `UNSURE`；此时应重新站直、调整取景或重新两点标定。

“显示手指节点”“隐藏面部”和“镜像画面”彼此同级，也与“骨架设置”分开。仅上半身、仅下半身本身会限制可见节点范围，因此不在所选范围内的面部或手指节点不会显示。

### 逐次动作语音提示

- 语音开关默认开启，选择会保存在当前浏览器中；
- 一次人体规则候选或分析周期完成时，系统汇总该次动作的改进项并立即播报，最多两条；
- 播报内容与文字报告“逐次动作表现”的汇总规则一致，包含持续性动作问题和清晰关键阶段的角度偏离；
- 如果本次没有持续性问题，状态会显示“未发现持续性问题”，但不会播放语音；
- 新动作建议到达时会替换过期的语音队列，避免播报明显滞后；暂停、停止、切换动作或离开页面时会取消正在等待的语音；
- 农夫行走属于连续动作：问题持续约 1.2 秒后播报，同一问题至少间隔 8 秒才会重复提示；
- 语音由浏览器 Web Speech API 在本机合成。系统优先选择中文语音，实际音色由操作系统和浏览器决定。

语音功能同时适用于本机摄像头、示例视频和上传视频。它只朗读需要改进的地方，不朗读整份文字报告。

## 3. 示例视频与上传视频

切换画面来源后可以选择示例视频或上传本地视频。上传限制为：

- 单个文件最大 250 MB；
- 每个匿名会话临时空间最大 500 MB；
- 服务端会检查实际视频内容，不只检查文件扩展名。

上传视频只作为临时分析输入，分析结束后立即删除；未分析的临时文件会随会话过期清理。用户不能读取其他会话的上传文件、状态或输出。

## 4. 数据、隐私与下载

- **摄像头帧**：通过加密公网连接发送给推理服务，只用于实时分析，不写入服务器磁盘。
- **语音提示**：“动作反馈”区域可开启或关闭语音。每完成人体规则候选或分析周期后，浏览器会在本机播报该段最需要改进的内容；农夫行走等连续动作会在问题持续出现后提示，并自动限制重复频率。
- **语音隐私**：语音由浏览器根据文字建议在本机合成，不使用麦克风，也不会向服务器上传音频。
- **麦克风**：不申请、不采集。
- **服务器录制**：已关闭，不生成或保存原始/标注视频。
- **截图**：只有用户主动点击时才生成，并直接下载到当前设备，服务器不保存。
- **分析报告**：可下载文字报告、JSON 或 CSV；停止分析后最多保留 10 分钟。
- **立即删除**：点击“删除本次数据”可清除当前匿名会话的上传文件和分析结果。

文字报告按总体结论、优先改进和逐次动作表现组织；JSON 包含逐帧关键点、动作阶段、人体规则计数或分析周期、反馈和性能数据；CSV 适合 Excel 或其他表格工具。三种报告都不包含原始或标注视频画面。分析距离动作报告时应优先读取 `cycle_count` 和 `count_semantics`，不要仅根据通用 `rep_count` 字段推断官方次数。

JSON 和 CSV 报告统一包含 `schema_version` 与 `program_version`，JSON 还包含
`artifact_type`。没有 schema 的旧文件按版本 `0` 处理；高于当前程序支持版本的文件
会以 `SCH001` 拒绝读取，应升级程序而不是手工猜测字段。

## 5. 浏览器与权限排查

推荐使用较新的 Chrome、Edge 或 Safari。实体设备仍应分别验收 Windows Chrome/Edge、macOS Safari、iPhone Safari 和 Android Chrome。

### 语音开关显示“语音不可用”或没有声音

- 优先使用较新的 Chrome、Edge 或 Safari，并确认系统媒体音量未静音；
- 如果显示“语音不可用”，表示浏览器未提供 Web Speech API，此时文字反馈和报告仍可正常使用；
- 某些移动浏览器要求先有一次用户点击才能播放语音，请直接点击一次“语音关”，再点击恢复“语音开”；
- 页面切到后台、暂停或停止分析时会主动取消语音，这是预期行为；
- 语音使用扬声器输出，不需要开启网站麦克风权限。

### 无法开启摄像头

- 确认正在使用 `http://127.0.0.1`、`http://localhost` 或 HTTPS 地址；
- 点击地址栏旁的网站设置，将“摄像头”改为允许后刷新页面；
- 关闭会议软件、系统相机等可能占用摄像头的程序；
- iPhone/iPad 请使用 Safari，并确认系统设置未禁止浏览器摄像头；
- 若设备列表为空，检查系统摄像头权限和硬件连接。

### 有预览但没有分析结果

- 点击“开始实时分析”，确认页面显示已连接；
- 让人物全身进入画面，并提高光照；
- 确认选择的动作和实际动作一致；
- 在高级设置中切换纯 MediaPipe、YOLO + MediaPipe 或 RTMW WholeBody，也可调整灵敏度；
- 网络较慢时系统会降低发送帧率，短暂等待连接自动恢复。

### 结果一直是“无法确认”

- 确保决定性人体部位全程入镜，并避免手腕、膝盖、脚跟或脚尖被器械和其他人遮挡；
- 确认拍摄视角与动作建议一致；已知不合适的视角会主动降级为 `UNSURE`；
- 等待系统自动建立绿色地板线，或在固定相机后使用两点标定；
- 快速动作中若关键端点被网络丢帧，可使用“中/高”灵敏度，但不要把无法确认手工改记成未完成。

### 公网地址打不开

- 保持 `启动公网访问.bat` 的窗口处于打开状态；
- 重新启动后使用窗口中新生成的地址，旧地址已经失效；
- 检查防火墙、代理或所在网络是否阻止 Cloudflare Tunnel；
- Quick Tunnel 无法保证中国大陆网络的稳定性，正式使用需要固定部署。

## 6. 容量与环境变量

默认接受最多 100 个匿名网页会话、最多 50 个实时分析会话。它们是保护服务的容量上限，不代表任意电脑都能让 50 人流畅推理；正式上线前必须在目标服务器完成 2、5、10、50 会话和 30 分钟压力测试。

```powershell
$env:POSE_MAX_SESSIONS = "100"
$env:POSE_MAX_ACTIVE = "50"
$env:POSE_MAX_SESSIONS_PER_IP = "10"
$env:POSE_INFERENCE_CONCURRENCY = "1"
$env:POSE_OUTPUT_DIR = "D:\pose-outputs"
.venv\Scripts\python.exe start_web.py
```

`POSE_OUTPUT_DIR` 只改变网页会话、上传、报告以及 `pose-doctor` / `pose-clean`
使用的输出根目录。网页启动日志默认仍位于当前目录的 `outputs/logs/web.log`；
需要改变日志位置或记录 traceback 时使用：

```powershell
pose-web --log-dir "D:\pose-logs" --debug --no-browser
```

日志单个文件最多 2 MiB，保留 5 个备份。磁盘清理默认仅预览，不会直接删除：

```powershell
pose-clean --json
pose-clean --older-than-days 30 --max-total-gb 20
pose-clean --older-than-days 30 --max-total-gb 20 --apply
```

只有最后一条带 `--apply` 的命令会执行删除；清理器只处理已登记的生成目录，不会删除
模型、配置或输入视频。

无 GPU 时可使用 CPU；RTMW-X/L 在 CPU 上精度不变但帧率会明显降低，实时手机访问建议部署在 NVIDIA GPU 服务器。服务器安装方式和官方模型地址见 [models/README.md](models/README.md)。手机只负责采集、压缩和绘制，RTMW 推理由服务器完成，但手机性能、摄像头和网络仍会影响采集帧率、清晰度、发热和端到端延迟。

## 7. 正式部署要求

当前 `start_web.py` 使用 Flask 本地服务，`start_public_web.py` 使用临时 Quick Tunnel。正式公网环境至少还需要：

- 长期有效的域名和 HTTPS；
- 支持 WebSocket 的生产 WSGI/ASGI 服务或反向代理；
- 进程守护、自动重启、健康检查、日志和告警；
- 目标地区的网络可达性与合规方案；
- 实体设备兼容测试、多用户隔离复测和容量压力测试。
