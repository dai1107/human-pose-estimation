# MediaPipe 产品主链路、三维坐标增强与实时低延迟优化实施方案

请在当前 HYROX 姿态分析项目基础上，完成一轮面向正式产品主链路的架构收敛与实时优化。

方案制定时项目支持 MediaPipe、YOLO + MediaPipe、YOLO + RTMW 三种姿态方案，动作分析主要基于二维关键点、One Euro 平滑、动作状态机、触地与脚部事件完成。2026-07-21 实施后，正式产品链路已统一为 MediaPipe；其他后端仅保留为实验和离线比较能力。

本轮需要在已有摄像头低延迟方案基础上，将正式产品运行链路统一为 MediaPipe，并接入 MediaPipe World Landmarks 作为三维运动学增强，同时解决摄像头帧积压、骨架错位、滤波拖尾、网页请求堆积和显示阻塞等问题。

## 实施状态（2026-07-21）

本方案的产品后端收敛、最新帧流水线、真实时间 One Euro、3D Shadow/Assist、桌面入口模块化、网页事件驱动结果推送和固定示例缓存均已落地。当前正式配置为：MediaPipe Pose、`latest_frame_only: true`、单请求在途、640 px 推理长边、JPEG 0.65、`responsive` 平滑、3D `assist`。

本机 30 帧网页协议探针测得往返延迟 P50 `18.7 ms`、P95 `35.5 ms`，原轮询链路约 `58 ms`；服务器处理 P50 `17.2 ms`、姿态推理 P50 `14.4 ms`、姿态年龄 P50 `16.0 ms`。固定箭步蹲示例 133 帧的无播放节流计算基准约 `1.1 s`、运行时推理 `0.0 ms`，原链路约 `15.7 s`；产品页面按源视频 30 FPS 展示，实际保持约 `4.43 s` 的原始时长。这些数据只代表当前测试机和本机协议环境。

8 个内置示例均使用带视频/模型指纹和 schema 校验的 v2 姿态、world landmarks 与手部缓存。箭步蹲缓存的 `yolo-guided-mediapipe` 仅表示离线生成及 image/world 身份匹配来源，产品页面显示“预计算示例结果”；缓存失效时自动回退真实推理。离线黄金回归始终绕过示例缓存，以保证模型和动作算法仍被真实验证。

# 一、总体目标

最终产品主链路统一为：

```text
摄像头最新帧
    ↓
MediaPipe Pose Landmarker
    ↓
2D image landmarks + 3D world landmarks
    ↓
时序平滑与质量检测
    ↓
二维特征 + 三维增强特征
    ↓
HYROX 动作状态机和规则系统
    ↓
有效计数、技术反馈、骨架绘制和语音提示
```

必须同时满足以下三个目标。

## 1. 产品主链路统一 MediaPipe

正式摄像头分析、网页实时分析和默认桌面运行只使用 MediaPipe Pose。

YOLO Pose 和 RTMW 不再参与正式产品实时链路，但暂时不要立即删除全部代码，可以保留为：

```text
experimental
benchmark_only
offline_comparison
research_ablation
```

它们只用于：

* 离线模型精度比较；
* 论文消融实验；
* 已录制视频 benchmark；
* 未来必要时重新评估。

## 2. 引入三维坐标增强

同时保留两套 MediaPipe 输出：

```text
pose_landmarks
    图像空间 x、y、相对 z、visibility、presence

pose_world_landmarks
    人体相对三维坐标 x、y、z
```

其中：

* 二维图像坐标继续负责地板、触地、起跳、落地、手腕高度和画面位移；
* 三维世界坐标主要增强膝、髋、肘、肩和躯干关节角度；
* 三维结果不可靠时自动回退二维；
* 不把单目估计的三维坐标解释成精确厘米距离。

## 3. 实时低延迟

实时模式采用：

```text
最新帧优先
单任务在途
忙时丢弃旧输入
过期结果拒绝
视频显示与姿态推理解耦
```

不再补算历史摄像头帧。

# 二、本轮不做的事情

本轮不要：

* 重新训练 MediaPipe Pose 模型；
* 删除 HYROX 动作状态机；
* 将所有二维规则直接替换成三维规则；
* 使用三维坐标判断精确地面距离；
* 改变 `VALID`、`NO_REP`、`UNSURE` 的含义；
* 改变现有动作阶段顺序；
* 直接套用原二维角度阈值判断三维角度；
* 让预测关键点进入正式计数状态机；
* 为提高显示 FPS 而把旧骨架无提示地画到当前帧；
* 破坏离线视频分析的确定性；
* 在本轮引入新的大型神经网络。

本轮优化的是：

> MediaPipe 的运行方式、坐标利用方式、关键点质量、时序处理和实时流水线，而不是重新训练 MediaPipe 模型本体。

# 五、统一实时帧和 MediaPipe 姿态结构

不要继续只传递裸图像或没有时间信息的 landmarks。

## 5.1 摄像头帧

```python
from dataclasses import dataclass
import numpy as np


@dataclass(slots=True)
class CapturedFrame:
    frame_id: int
    capture_timestamp_ns: int
    image: np.ndarray
    source: str
    width: int
    height: int
```

要求：

* 每个实时会话的 `frame_id` 严格递增；
* 摄像头成功读帧后立即生成 `capture_timestamp_ns`；
* 使用 `time.perf_counter_ns()`；
* 镜像、缩放和编码不应覆盖原始采集时间；
* 离线视频继续使用视频时间轴，不强制使用真实墙钟时间。

## 5.2 姿态关键点结构

建议新增：

```python
@dataclass(slots=True)
class PoseLandmark:
    x: float
    y: float
    z: float | None
    visibility: float | None
    presence: float | None


@dataclass(slots=True)
class PoseLandmarkSet:
    image_landmarks: tuple[PoseLandmark, ...]
    world_landmarks: tuple[PoseLandmark, ...] | None
```

不要把 MediaPipe 的 image landmarks 和 world landmarks 混成一套坐标。

## 5.3 带时间的姿态结果

```python
@dataclass(slots=True)
class TimedPoseResult:
```

# 六、MediaPipe 三维坐标增强架构

## 6.1 同时保留二维和三维

每次 MediaPipe callback 返回时，同时提取：

```python
result.pose_landmarks
result.pose_world_landmarks
```

统一输出：

```text
image_landmarks[33]
world_landmarks[33] 或 None
```

若 world landmarks 缺失：

* 不报错；
* 继续二维处理；
* 将三维可用状态记为 `false`。

## 6.2 不覆盖现有二维角度

现有角度字段继续保留：

```text
left_knee_angle_2d
right_knee_angle_2d
left_hip_angle_2d
right_hip_angle_2d
left_elbow_angle_2d
right_elbow_angle_2d
```

新增：

```text
left_knee_angle_3d
right_knee_angle_3d
left_hip_angle_3d
right_hip_angle_3d
left_elbow_angle_3d
right_elbow_angle_3d
left_shoulder_angle_3d
right_shoulder_angle_3d
```

三维角度计算：

```python
def calculate_angle_3d(
    point_a: np.ndarray,
    vertex: np.ndarray,
    point_b: np.ndarray,
) -> float | None:
    vector_a = point_a - vertex
    vector_b = point_b - vertex

    norm_a = np.linalg.norm(vector_a)
    norm_b = np.linalg.norm(vector_b)

    if norm_a <= 1e-8 or norm_b <= 1e-8:
        return None

    cosine = np.dot(vector_a, vector_b) / (norm_a * norm_b)
    cosine = float(np.clip(cosine, -1.0, 1.0))
    return float(np.degrees(np.arccos(cosine)))
```

必须处理：

* NaN；
* 无限值；
* 零长度向量；
* world landmarks 缺失；
* 低可见度节点。

## 6.3 增加三维可靠性评估

不能因为存在 `z` 就直接使用三维角度。

建议新增：

```python
@dataclass(slots=True)
class AngleMeasurement:
    angle_2d: float | None
    angle_3d: float | None
    selected_angle: float | None
    selected_source: str
    confidence: float
    three_d_reliable: bool
```

三维可靠性至少检查：

1. 三个关节均存在；
2. 坐标是有限值；
3. image landmarks 的 visibility 达标；
4. presence 达标；
5. 相邻帧骨段长度没有异常跳变；
6. z 没有突然发生不合理跳变；
7. 三维角速度没有明显超出生理可能范围；
8. 左右肢体没有发生身份交换；
9. 当前结果没有超过最大姿态年龄；
10. 当前帧与上一三维帧间隔没有超过重置阈值。

初始建议：

```yaml
three_d_quality:
  min_visibility: 0.70
  min_presence: 0.70
  max_bone_length_change_ratio: 0.20
  max_angle_delta_deg: 35.0
  max_2d_3d_difference_deg: 25.0
  max_gap_ms_before_reset: 250
```

这些只是初始值，需要通过视频数据标定。

## 6.4 三维分阶段启用

不要第一轮就让三维影响计数。

### 阶段 A：Shadow Mode

计算并保存三维角度，但：

* 状态机仍只使用二维；
* 计数仍只使用二维；
* 报告增加二维与三维差值；
* 收集不同动作、视角和阶段的数据。

配置：

```yaml
three_d_kinematics:
  enabled: true
  decision_mode: shadow
```

### 阶段 B：Assist Mode

三维只作为置信度增强证据。

例如：

```text
二维和三维都满足伸展要求：
    提高规则置信度

二维接近阈值，三维明显通过：
    保持候选，但不要直接改为 VALID

二维和三维严重冲突：
    降低置信度或标记 UNSURE
```

此阶段仍不直接改变现有计数阈值。

### 阶段 C：Rule-specific Mode

完成数据标定后，才允许特定规则正式使用三维。

必须分别设置：

```yaml
knee_extension_threshold_2d_deg: 165
knee_extension_threshold_3d_deg: 168
```

禁止让二维和三维共享一个未经验证的阈值。

# 七、二维和三维的职责划分

## 7.1 优先使用三维增强的特征

三维适合：

* 膝关节屈伸；
* 髋关节屈伸；
* 肘关节屈伸；
* 肩关节角度；
* 躯干与大腿夹角；
* 斜侧拍摄下的肢体伸展程度；
* 左右肢体的空间不对称。

## 7.2 继续使用二维图像坐标的特征

以下规则继续以二维为主：

* 后膝是否接近地板；
* 胸部代理是否触地；
* 髋部是否低于膝部；
* 手腕是否高于肩；
* 双脚是否离地；
* 起跳和落地时间；
* 脚部前后错位；
* 落地后是否补步；
* 人体是否向前移动；
* 人体是否出画；
* 地板线和局部离地高度。

MediaPipe world landmarks 是人体相对三维估计，不能直接用于：

```text
膝盖距离地板 3 厘米
胸部距离地板 2 厘米
跳远距离 1.5 米
```

# 九、桌面版低延迟流水线

## 9.1 摄像头独立线程

新增：

```python
class LatestFrameCamera:
    ...
```

要求：

* 独立线程持续读取摄像头；
* 缓冲容量固定为 1；
* 新帧覆盖旧帧；
* 不创建历史帧队列；
* 摄像头读取不等待姿态推理；
* 退出时正确释放；
* 读取失败时适当休眠，避免忙循环。

尝试设置：

```python
cap.set(cv2.CAP_PROP_BUFFERSIZE, 1)
```

保留：

```text
Windows 优先 CAP_DSHOW
失败后回退普通 VideoCapture
```

不假设驱动一定支持缓冲区设置。

## 9.2 最新帧优先

主循环每次只取最新帧：

```text
摄像头帧 100、101、102、103 到达
MediaPipe 正在处理 100
完成后直接处理 103
不再补处理 101 和 102
```

这属于正常实时丢帧，不是错误。

## 9.3 单任务在途

新增：

```python
class LatestOnlyMediaPipeScheduler:
    ...
```

规则：

* 同一会话最多一个 MediaPipe 请求在途；
* 后端空闲时提交最新帧；
* 后端忙碌时只更新最新待处理帧；
* callback 完成后立即提交当时最新帧；
* 不补算历史帧；
* 所有状态释放放在 `finally`；
* 关闭后迟到 callback 不得访问已释放对象。

## 9.4 callback 帧身份

MediaPipe 提交时建立：

```python
timestamp_ms -> CapturedFrame metadata
```

callback 返回后根据 MediaPipe 时间戳恢复：

```text
frame_id
capture_timestamp_ns
inference_start_ns
```

不要仅根据 callback 到达顺序猜测帧身份。

MediaPipe 时间戳必须严格递增，即使多个帧落在同一毫秒。

可使用：

```python
timestamp_ms = max(
    capture_timestamp_ns // 1_000_000,
    previous_timestamp_ms + 1,
)
```

# 十、网页低延迟流水线

## 10.1 视频显示与推理解耦

浏览器 `<video>` 始终直接显示摄像头画面。

服务器姿态推理不得阻塞视频显示。

结构：

```text
浏览器原始视频：持续显示
推理请求：较低频率发送最新帧
骨架：绘制最近未过期结果
```

## 10.2 单请求在途

浏览器实时会话最多一个姿态请求在途。

```javascript
let requestInFlight = false;
let latestFrameId = 0;
let lastRenderedPoseFrameId = -1;
```

发送逻辑：

```text
没有请求在途
    → 抓取当前最新视频帧
    → 编码
    → 发送
    → 等待响应
    → 收到或超时
    → 立即发送此刻最新帧
```

禁止使用无门控的 `setInterval()` 连续积压请求。

## 10.3 请求帧结构

请求至少携带：

```json
{
  "session_id": "...",
  "frame_id": 123,
  "client_capture_ms": 45678.1,
  "action": "lunge",
  "backend": "mediapipe"
}
```

响应：

```json
{
  "session_id": "...",
```

# 十一、姿态结果过期门控

建议配置：

```yaml
realtime_latency:
  warning_pose_age_ms: 80
  max_pose_age_ms: 150
  max_frame_gap: 5
  hide_pose_after_ms: 300
```

以下情况视为过期：

* 结果年龄超过 `max_pose_age_ms`；
* 当前显示帧与姿态帧差距超过 `max_frame_gap`；
* 结果属于旧会话；
* 结果属于已停止的摄像头；
* 结果属于旧动作；
* 结果属于旧后端；
* callback 返回顺序落后于已经接受的结果。

过期结果：

* 不进入动作分析；
* 不覆盖当前姿态；
* 增加 stale drop 指标；
* 可以短暂显示最后一个未过期姿态并降低透明度；
* 超过隐藏时间后停止绘制。

动作分析器只接受：

```python
result.frame_id > last_analyzed_frame_id
```

# 十三、二维和三维滤波策略

不要直接对最终角度做强平滑。

推荐顺序：

```text
原始关键点
    ↓
关键点质量门控
    ↓
2D 关键点 One Euro
3D world landmarks One Euro
    ↓
分别计算 2D 和 3D 角度
    ↓
角度可靠性和冲突检查
```

原因：

* 对关键点滤波可以保持骨段关系；
* 对角度再次重滤波容易增加动作端点延迟；
* 三维 z 通常比 x、y 更容易抖动，可以为 z 使用略稳定的参数；
* 快速动作端点不能被过强平滑抹掉。

# 第 6 轮实施记录（2026-07-21）

本轮已完成 One Euro 低延迟改造，未改变 HYROX 动作阶段、阈值以及 `VALID`、`NO_REP`、`UNSURE` 语义。

实现结果：
