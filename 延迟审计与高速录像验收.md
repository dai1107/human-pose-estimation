# 实时姿态完整延迟审计与高速录像验收

本说明对应《实时姿态显示延迟优化》的第一阶段。此阶段只测量链路，不改变姿态模型、平滑参数或动作判定。

## 内部审计

网页实时摄像头会自动采集视频帧呈现、画面拷贝、JPEG 编码、WebSocket、服务端接收、推理、客户端接收和首次骨架绘制时刻。下载页面 JSON 或 CSV 报告即可保留逐帧数据；JSON 的 `summary.latency_audit` 是 P50/P95 汇总，逐帧原始数据位于 `latency_timing`，派生数据位于 `latency`。

浏览器和服务器使用不同的单调时钟。报告只在同一时钟域内相减：完整网络往返使用浏览器的 `socket_send_ms` 到 `client_result_receive_ms`，服务端时间戳用于拆分服务端内部耗时，绝不直接与浏览器时间戳相减。

桌面版默认在退出时写入：

```text
outputs/benchmarks/desktop_latency_audit.json
```

可用 `--latency-audit 路径` 改名，传空字符串关闭落盘。OpenCV 无法提供显示器真正的合成截止时间，因此桌面版 `render_to_expected_display_ms` 是 `draw_end_ns` 到 `imshow_return_ns` 的可测提交代理；真实显示器延迟必须用下面的高速录像补齐。

统一报告的核心指标是 `pose_video_age_difference_ms`。它表示正在绘制的姿态对应帧比当前视频帧老多少，而不是只看模型推理耗时。

## 120/240 FPS 外部验收

1. 手机选择固定的 120 FPS 或 240 FPS 慢动作模式，关闭会动态改变帧率的省电功能。
2. 同一画面同时拍到测试者的手和完整电脑屏幕，尽量避免滚动快门方向与手的运动方向一致。
3. 测试者快速左右摆手，或从静止状态快速拍击桌面；每个配置至少记录 10 次。
4. 逐帧找到真实手首次明确移动的帧号，以及屏幕骨架首次同方向移动的帧号。
5. 使用工具计算并保存结果：

```powershell
pose-latency-benchmark --mode web-server `
  --input web-report.json `
  --recording-fps 240 `
  --physical-motion-frame 100 `
  --display-motion-frame 112 `
  --output outputs/benchmarks/web-server-240fps.json
```

上例外部 sensor-to-photon 延迟为 `(112 - 100) / 240 × 1000 = 50 ms`。

若源码尚未安装为命令，也可以运行：

```powershell
python -m tools.pose_latency_benchmark --mode desktop --input outputs/benchmarks/desktop_latency_audit.json
```

## 最小测试矩阵

分别保存网页版 MediaPipe、桌面版 MediaPipe，以及关闭平滑和 `responsive` 平滑的结果。摄像头规格至少覆盖 640×480 30 FPS、640×480 60 FPS、1280×720 30 FPS。显示预测尚未实施时应明确记为“未启用”，不得用内部推理时间代替外部高速录像结果。

每次结果至少记录设备、浏览器或 OpenCV 版本、摄像头实际规格、光照、平滑设置、内部报告路径、慢动作 FPS、两处事件帧号和计算后的外部延迟。
