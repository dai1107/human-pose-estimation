# HYROX 公网多人摄像头网页版任务清单

## 1. 目标与当前状态

### 最终目标

- [ ] 用户只需打开一个长期有效的 HTTPS 网址即可使用。
- [ ] 支持 Windows、macOS、Android 和 iPhone 的主流浏览器。
- [ ] 用户可授权并使用自己设备的摄像头进行实时姿态分析。
- [ ] 用户也可上传本地视频进行分析。
- [ ] 每位用户拥有独立分析会话，互不干扰。
- [ ] 未经用户明确授权时，网页不得读取摄像头。
- [ ] 默认不采集麦克风音频。
- [ ] 提供清晰的隐私说明、错误提示和停止摄像头入口。

### 当前已经具备

- [x] Flask 网页前端和分析接口。
- [x] HYROX 动作选择、视角、灵敏度和模型选择。
- [x] 访客浏览器主动授权并使用自己设备的摄像头。
- [x] WebSocket 实时传帧、最新帧队列、服务端推理和 Canvas 本地骨架绘制。
- [x] 用户本地视频上传分析。
- [x] 匿名随机会话、每用户独立分析状态、上传目录和报告。
- [x] 摄像头帧不落盘、本地截图、服务器录制关闭和 10 分钟 JSON/CSV 报告。
- [x] 250 MB 单文件限制、500 MB 单会话配额、真实媒体校验和过期清理。
- [x] CSRF、WebSocket 来源校验、速率限制、安全响应头和容量保护。
- [x] 临时公网 HTTPS 链接。
- [x] 默认匿名访问，可选随机访问口令保护。
- [x] 电脑和手机自适应页面布局。

### 当前主要限制

- [ ] Cloudflare Quick Tunnel 地址重启后变化，不适合作为正式长期网址。
- [ ] Flask 开发服务器不适合作为正式公网生产服务器。
- [ ] 尚无长期域名、目标服务器、生产进程守护、监控告警和中国大陆稳定入口。
- [ ] 尚未完成 Windows/macOS/iPhone/Android 实体设备人工兼容验收。
- [ ] 尚未完成 2、5、10、50 个真实会话和 30 分钟压力测试。
- [ ] 当前按已确认方案采用匿名临时会话，不建设账户或登录系统；正式运营前仍需补充完整隐私政策和合规评估。

## 2. 推荐技术路线

建议采用“浏览器采集摄像头 + 服务端姿态推理”的第一版方案：

1. 浏览器通过 `navigator.mediaDevices.getUserMedia()` 请求当前用户的摄像头权限。
2. 浏览器只采集视频，不请求麦克风。
3. 浏览器按设定帧率和分辨率压缩画面，通过 WebSocket 发送到服务器。
4. 服务器为每个用户创建独立会话并执行 MediaPipe 或 YOLO Pose 推理。
5. 服务器返回关键点、动作阶段、次数、提示和必要的标注画面。
6. 浏览器在本地显示原始视频，并使用 Canvas 叠加骨架和反馈，减少视频回传带宽。

暂不建议第一版直接把完整 Python 模型改写成单个离线 HTML。现有分析逻辑、配置和模型均在 Python 后端，强行改成纯前端会形成一套新的实现，验证成本较高。

## 3. P0：需求冻结与技术验证

- [ ] 明确是否允许匿名访问，还是必须登录后使用。

### 3.2 浏览器摄像头验证原型

- [ ] 新建最小摄像头测试页，只请求 `{ video: true, audio: false }`。
- [ ] 支持前置和后置摄像头切换。
- [ ] 显示摄像头授权中、已允许、已拒绝和设备不存在状态。
- [ ] 用户点击“停止”后立即关闭所有 `MediaStreamTrack`。
- [ ] 页面切换到后台或关闭时释放摄像头。
- [ ] 在 Chrome、Edge、Safari 和 Android 浏览器上验证。
- [ ] 测试 HTTPS 环境下的摄像头权限。

### 3.3 传输方案验证

- [ ] 建立 WebSocket 连接原型。
- [ ] 测试 JPEG/WebP 帧传输，初始目标为 640×480、10–15 FPS。
- [ ] 测试不同网络条件下的延迟和丢帧处理。

- [ ] 断线后允许自动重连并恢复当前会话状态。
- [ ] 记录端到端延迟、上传带宽、推理耗时和浏览器 CPU 占用。

### P0 验收标准

- [ ] 手机和电脑均能授权自己的摄像头并看到预览。
- [ ] 服务端能连续收到帧并返回一次真实姿态识别结果。

- [ ] 点击停止或关闭页面后摄像头指示灯熄灭。
- [ ] 典型网络下端到端反馈延迟目标不超过 500 ms。

## 5. P1：多用户会话隔离

### 5.1 会话模型

- [ ] 不再使用全局单例 `PoseStreamEngine`。
- [ ] 为每位访问者生成不可预测的 `session_id`。
- [ ] 建立 `session_id -> engine/state/queue` 映射。
- [ ] 每个会话拥有独立的动作、视角、灵敏度和模型设置。
- [ ] 每个会话拥有独立的上传文件和输出目录。
- [ ] 一个用户的停止、截图和录制操作不能影响其他用户。
- [ ] 连接断开后进入宽限期，超时自动释放会话资源。
- [ ] 服务重启后不恢复摄像头会话。

### 5.2 并发与资源控制

- [ ] 设置最大同时分析会话数。
- [ ] 设置每个 IP 或账户的会话数限制。
- [ ] GPU 推理使用有界任务队列。
- [ ] 超出容量时返回“服务器繁忙”，而不是拖垮所有会话。
- [ ] 自动降低低性能设备的分辨率或 FPS。
- [ ] 记录 CPU、GPU、显存、内存、队列长度和会话数。
- [ ] 压力测试 2、5、10 个并发会话。

### 5.3 临时数据清理

- [ ] 上传文件按会话保存。
- [ ] 验证文件扩展名之外的真实媒体类型。
- [ ] 设置单文件和单用户空间配额。
- [ ] 定时清理过期上传、截图、录制和分析结果。
- [ ] 防止用户通过路径参数访问其他人的文件。
- [ ] 输出下载地址使用短期有效的签名令牌。

### 多用户验收标准

- [ ] 两台手机同时分析时状态、画面和次数完全隔离。
- [ ] 用户 A 无法停止、读取或下载用户 B 的任务和文件。
- [ ] 用户异常断网后资源可自动释放。
- [ ] 达到并发上限时已有会话仍然稳定。

## 6. P1：安全与隐私

### 6.1 身份与访问控制

- [ ] 决定采用匿名临时会话、邀请码或完整账户系统。
- [ ] 正式环境不再把长期访问令牌放在 URL 查询参数中。
- [ ] 登录 Cookie 设置 `HttpOnly`、`Secure` 和合适的 `SameSite`。
- [ ] 所有修改状态的 HTTP 接口加入 CSRF 防护或采用等效令牌机制。
- [ ] WebSocket 握手校验来源、会话和身份。
- [ ] 配置请求速率限制和失败重试限制。
- [ ] 禁止生产环境调试模式和详细异常页面。

### 6.2 摄像头与隐私

- [ ] 在首次开启摄像头前显示用途说明。
- [ ] 明确声明是否上传视频帧、保存多久、谁能访问。
- [ ] 默认不采集麦克风。
- [ ] 未开启录制时不保存原始摄像头帧。
- [ ] 录制和截图必须由用户主动触发并显示明显状态。
- [ ] 提供删除当前会话数据的入口。
- [ ] 页面离开和会话结束时停止摄像头并释放服务器资源。
- [ ] 编写隐私政策和用户使用说明。

### 6.3 服务端防护

- [ ] 配置 HTTPS、可信代理头和安全响应头。
- [ ] 配置 `Content-Security-Policy`。
- [ ] 配置 `Permissions-Policy: camera=(self), microphone=()`。
- [ ] 配置 `X-Content-Type-Options: nosniff`。
- [ ] 限制跨域来源，不使用宽泛 `*`。
- [ ] 上传文件不放在可直接执行或公开枚举的目录。
- [ ] 锁定 Python 依赖版本并定期检查漏洞。
- [ ] 日志不得记录访问令牌、Cookie 或完整摄像头帧。

### 安全验收标准

- [ ] 未授权用户不能读取会话状态、视频流或输出文件。
- [ ] 拒绝摄像头权限后网页仍可使用上传视频功能。
- [ ] 页面不会请求麦克风权限。
- [ ] 常见恶意上传、超大请求和高频请求会被拒绝。
- [ ] 安全响应头和 Cookie 属性通过自动化测试。

## 7. P2：正式公网部署

### 7.1 部署形式

- [ ] 选择长期运行的服务器：云 GPU、云 CPU 或自有 GPU 主机。
- [ ] 根据并发测试结果确定 CPU、内存、GPU 和带宽规格。
- [ ] 使用固定域名，例如 `pose.example.com`。
- [ ] 配置正式 Cloudflare Tunnel、反向代理或云负载均衡。
- [ ] 使用生产 WSGI/ASGI 服务，不使用 Flask 自带开发服务器。
- [ ] 配置进程守护、自动重启和开机启动。
- [ ] 配置健康检查和滚动更新。
- [ ] 将密钥和数据库凭据放入环境变量或密钥管理服务。

### 7.2 存储与数据库

- [ ] 用户和会话元数据保存到数据库。
- [ ] 大视频和报告使用对象存储，不直接长期堆积在应用磁盘。
- [ ] 设置自动过期策略。
- [ ] 如需保留用户数据，建立备份和恢复流程。
- [ ] 数据库迁移纳入版本控制和部署流程。

### 7.3 可观测性

- [ ] 收集请求错误率、响应延迟和 WebSocket 断开率。
- [ ] 收集每个模型的推理延迟和失败率。
- [ ] 设置服务器不可用、磁盘不足、GPU 异常和高错误率告警。
- [ ] 日志带请求 ID 和会话 ID，但不记录敏感内容。
- [ ] 建立基本运行状态页面。

### 正式部署验收标准

- [ ] 域名长期有效，服务重启不改变访问网址。
- [ ] 服务器重启后应用自动恢复。
- [ ] HTTPS 证书自动续期。
- [ ] 外部网络的手机和电脑均可访问并使用自己的摄像头。
- [ ] 达到设计并发人数时性能满足目标。
- [ ] 监控、日志、告警、备份和数据清理均已验证。
