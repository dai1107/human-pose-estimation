# HYROX 实时姿态显示延迟优化与本地优先架构实施方案

## 一、任务背景

当前项目已经实现：

* MediaPipe Pose 产品主链路；
* 最新帧优先；
* 单任务在途；
* 忙时丢弃旧帧；
* WebSocket 事件驱动结果推送；
* 帧编号与采集时间关联；
* 过期姿态结果拒绝；
* 自适应 One Euro Filter；
* `stable`、`balanced`、`responsive` 三档平滑；
* 手指识别默认关闭；
* image landmarks 与 world landmarks 独立滤波；
* 30/60 分钟耐久测试；
* 487 项自动化测试。

现有网页协议测试结果已经显示：

```text
RTT P50：约 18.7 ms
RTT P95：约 35.5 ms
服务器处理 P50：约 17.2 ms
MediaPipe 推理 P50：约 14.4 ms
姿态年龄 P50：约 16.0 ms
```

但是实际观察时，人体关节节点仍明显落后于人体运动。

本轮不要重复实现已有的“最新帧”和“忙时丢帧”，重点解决：

1. 当前延迟指标没有包含完整的 sensor-to-photon 延迟。
2. 视频帧和骨架帧的显示时间没有严格关联。
3. One Euro 平滑导致快速运动拖尾。
4. 浏览器主线程和显示合成引入额外延迟。
5. 网页姿态推理仍经过图像编码、网络和服务器。
6. 缺少根据实际结果年龄进行短时间前向预测的显示层。
7. 摄像头内部缓冲、曝光和实际帧率尚未被系统评估。

# 二、总体优化目标

将系统拆分为两个不同目标的姿态链路。

## 2.1 权威动作分析链路

继续用于：

* HYROX 动作阶段识别；
* 触地判断；
* 起跳和落地判断；
* `VALID`、`NO_REP`、`UNSURE`；
* 技术问题识别；
* 报告和动作计数。

该链路必须使用真实观测结果，允许稳定滤波，但不得使用激进前向预测。

```text
真实姿态
    → 稳定型过滤
    → 运动学特征
    → 状态机
    → 计数和规则
```

## 2.2 低延迟骨架显示链路

只用于：

* 页面骨架显示；
* 关节圆点显示；
* 连接线显示；
* 用户实时视觉反馈。

该链路允许：

* 更弱的过滤；
* 动态原始坐标混合；
* 20～50 ms 的短时预测；
* 屏幕显示时间补偿。

```text
真实姿态
    → 低延迟过滤
    → 速度估计
    → 根据姿态年龄前向预测
    → 屏幕骨架
```

不得让显示预测结果直接进入正式动作状态机。

# 三、第一阶段：测量完整的 sensor-to-photon 延迟

> 实施状态（2026-07-23）：软件延迟审计已完成。网页记录摄像头呈现、
> 复制/编码、WebSocket、推理、结果接收、绘制和预计显示时间；桌面记录
> 读取、推理、分析、绘制和 `imshow` 返回时间。JSON/CSV 输出分段延迟、
> 姿态年龄、视频年龄及 `pose_video_age_difference_ms`，并提供
> `pose-latency-benchmark` 与 120/240 FPS 高速录像验收说明。摄像头曝光、
> 内部缓冲、显示器扫描等物理 sensor-to-photon 尚未在目标设备自动测量，
> 对应字段不得用软件耗时推算。

当前指标主要覆盖软件管线，没有完全覆盖摄像头和显示器。

先新增统一的端到端延迟审计，不要立即修改模型。

## 3.1 新增延迟分段

网页端至少记录：

```text
camera_frame_presented_ms
frame_copy_start_ms
frame_copy_end_ms
encode_start_ms
encode_end_ms
socket_send_ms
server_receive_ms
inference_start_ms
inference_end_ms
socket_result_send_ms
client_result_receive_ms
pose_render_start_ms
pose_render_end_ms
expected_display_time_ms
```

桌面端至少记录：

```text
capture_read_start_ns
capture_read_end_ns
inference_start_ns
inference_end_ns
analysis_end_ns
draw_start_ns
draw_end_ns
imshow_return_ns
```

## 3.2 区分四类延迟

输出：

```text
capture_to_submit_ms
submit_to_result_ms
result_to_render_ms
render_to_expected_display_ms
pose_age_at_render_ms
video_frame_age_at_render_ms
pose_video_age_difference_ms
```

其中最重要的指标是：

```text
pose_video_age_difference_ms
```

它代表骨架所对应帧和当前视频帧之间相差多少时间。

即使服务器只需要 16 ms，只要当前视频画面比姿态结果更新 40 ms，用户就会看到骨架明显落后。

## 3.3 增加外部高速录像验收

增加文档和工具支持真实 sensor-to-photon 验收。

测试方法：

1. 使用手机 120 FPS 或 240 FPS 慢动作录像。
2. 画面中同时拍摄用户的手和电脑屏幕。
3. 用户快速左右摆手或快速拍击桌面。
4. 逐帧计算真实手开始运动与屏幕骨架开始运动之间的差值。

建议至少测试：

```text
网页版 MediaPipe
桌面版 MediaPipe
关闭平滑
responsive 平滑
开启显示预测
关闭显示预测
640×480 30 FPS
640×480 60 FPS
1280×720 30 FPS
```

当前内部指标和高速录像结果都要保存，不能只报告服务器推理时间。

# 四、第二阶段：网页改用 requestVideoFrameCallback 驱动

> 实施状态（2026-07-23）：已完成视频帧驱动显示。支持的浏览器使用
> `requestVideoFrameCallback` 生成单调 `frameId`，采集提交、姿态选择和
> 骨架 Canvas 共用同一视频帧元数据；同时记录 `presentedFrames` 跳变、
> rVFC lateness 和预计显示时间。旧浏览器明确回退
> `requestAnimationFrame`，迟到帧和旧会话结果会被拒绝。独立 Canvas
> 继续作为默认显示方式；单 Canvas 未在缺少目标浏览器拷贝基准时擅自启用。

不要继续用普通 `setInterval` 或独立的 `requestAnimationFrame` 判断摄像头是否产生新帧。

使用：

```javascript
video.requestVideoFrameCallback(onVideoFrame);
```

该 API 会在新视频帧提交给浏览器合成器时回调，并可获得：

```text
presentationTime
expectedDisplayTime
mediaTime
presentedFrames
captureTime（浏览器支持时）
processingDuration
```

## 4.1 每个视频帧建立身份

定义：

```javascript
const frameMeta = {
  sessionId,
  frameId,
  presentedFrames,
  mediaTime,
  presentationTime,
  expectedDisplayTime,
  captureTime,
  width,
  height,
};
```

要求：

* `frameId` 单调递增。
* 记录 `presentedFrames` 跳变。
* 不再仅通过 `video.currentTime` 判断是否是新帧。
* 请求发送和骨架绘制均保留视频帧元数据。
* 浏览器不支持时再回退 `requestAnimationFrame`。

## 4.2 识别是否晚一个垂直同步周期

比较：

```javascript
const vsyncLatenessMs =
  performance.now() - metadata.expectedDisplayTime;
```

记录：

```text
rvfc_late_frame_count
rvfc_lateness_ms
presented_frame_skip_count
```

如果骨架绘制经常晚一个 60 Hz 垂直同步周期，会额外产生约一帧显示延迟。

## 4.3 避免视频与骨架使用两个独立时钟

当前如果：

* `<video>` 由浏览器合成线程显示；
* 骨架 canvas 由普通 `requestAnimationFrame` 更新；

两者可能相差一个刷新周期。

改为：

```text
requestVideoFrameCallback
    → 更新当前视频帧元数据
    → 选择当前可用姿态
    → 计算预测姿态
    → 更新骨架 canvas
```

如实测仍不同步，再增加可选的单 Canvas 模式：

```text
同一个 Canvas：
    先 drawImage(video)
    再绘制 skeleton
```

单 Canvas 可以保证视频和骨架在同一次提交中合成，但会增加一次视频拷贝，因此必须通过基准测试决定是否默认开启。

# 五、第三阶段：将网页版 MediaPipe 移到浏览器本地

> 实施状态（2026-07-23）：已完成浏览器本机 MediaPipe Worker。
> `pose_worker.js` 在模块 Worker 中运行 `PoseLandmarker`，主线程优先转移
> `VideoFrame`，否则转移 `ImageBitmap`；忙时覆盖唯一 pending 槽，不补算
> 历史帧。本机显示结果立即绘制，Python 只接收 Worker 的原始 image/world
> landmarks 执行 HYROX。Worker/WASM/模型或设备失败时仅在
> `server_pose_fallback` 允许的情况下明确回退；禁止回退时页面直接报错。

这是本轮最重要的架构优化。

现在网页摄像头链路大致是：

```text
摄像头
  → 浏览器 Canvas
  → JPEG 编码
  → WebSocket
  → Python
  → MediaPipe
  → 规则分析
  → WebSocket
  → 浏览器
  → 骨架绘制
```

建议改为：

```text
摄像头
  → 浏览器本地 MediaPipe
  → 立即绘制骨架
  → 只把关键点和时间戳发送给 Python
  → Python 执行 HYROX 规则
  → 返回反馈和计数
```

这样可以移除实时主链路中的：

* JPEG 编码；
* JPEG 解码；
* 视频帧上传；
* 服务器姿态模型；
* 姿态坐标返回等待。

## 5.1 浏览器姿态后端

使用：

```text
@mediapipe/tasks-vision
PoseLandmarker
runningMode: VIDEO
numPoses: 1
outputSegmentationMasks: false
```

正式摄像头使用本地 MediaPipe，服务器不再对同一帧重复运行 Pose Landmarker。

## 5.2 必须使用 Web Worker

MediaPipe 官方 Web API 的 `detectForVideo()` 是同步调用，会阻塞调用线程，因此不能直接放到页面主线程。

建立：

```text
webui/static/workers/pose_worker.js
```

主线程负责：

* 摄像头显示；
* `requestVideoFrameCallback`；
* 向 Worker 提交最新帧；
* 骨架绘制；
* 用户操作。

Worker 负责：

* 加载 MediaPipe；
* 运行 `detectForVideo()`；
* One Euro 或低延迟滤波；
* 返回 image landmarks；
* 返回 world landmarks；
* 返回推理耗时和输入帧编号。

## 5.3 Worker 采用最新帧槽位

Worker 忙时不建立普通消息队列。

主线程维护：

```javascript
let workerBusy = false;
let latestPendingFrame = null;
```

流程：

```text
Worker 空闲：
    发送当前帧

Worker 忙碌：
    用新帧覆盖 latestPendingFrame

Worker 返回：
    立即发送 latestPendingFrame
```

禁止逐帧补算。

## 5.4 避免昂贵的数据复制

优先评估以下方式：

1. `VideoFrame` 转移。
2. `ImageBitmap` 转移。
3. `OffscreenCanvas`。
4. 最后才使用普通像素数组复制。

所有能力必须经过特性检测，并提供兼容回退。

不要在主线程执行：

```javascript
canvas.toDataURL()
```

本地姿态链路不再需要 JPEG。

## 5.5 向服务器发送关键点

消息结构：

```json
{
  "type": "pose_frame",
  "session_id": "...",
  "frame_id": 1024,
  "capture_timestamp_ms": 12345.6,
  "presentation_timestamp_ms": 12358.9,
  "image_landmarks": [],
  "world_landmarks": [],
  "pose_inference_ms": 12.7,
  "source": "browser_mediapipe"
}
```

服务器继续执行：

* 2D 特征；
* 3D Assist；
* 地板；
* 接触检测；
* 脚部事件；
* 动作状态机；
* 完整周期验证；
* 语音反馈内容；
* 报告。

这样不需要重写现有 HYROX 分析器。

## 5.6 保留服务器姿态后端作为回退

以下情况回退当前服务器路径：

* 浏览器不支持 Web Worker 或相关 API；
* MediaPipe WASM 初始化失败；
* 设备性能不足；
* 用户显式选择兼容模式；
* 浏览器内存不足；
* 模型文件加载失败。

页面显示当前模式：

```text
本机实时姿态
服务器兼容姿态
```

不得静默回退。

# 六、第四阶段：增加 Lite、Full 两种产品模型档位

> 实施状态（2026-07-23）：Lite 与 Full 模型、约 3 秒设备基准和双模型
> 回归工具已完成。基准记录 P50/P95、姿态 FPS、检出率和主线程长任务。
> Full 仍是正式分析模型；Lite 原黄金区间仅 `3/8`、严格等价门仅 `1/8`
> 通过，因此 `lite_auto_approved=false`，自动档不会因速度静默切换 Lite。
> Lite 只保留为用户显式速度实验，正式黄金视频继续使用 Full 并通过 `8/8`。

当前正式模型文件是：

```text
pose_landmarker_full.task
```

增加：

```text
pose_landmarker_lite.task
pose_landmarker_full.task
```

不要默认引入 Heavy。

Google 的 Pose Landmarker 提供 Lite、Full 和 Heavy 模型变体。Lite 更适合低计算量实时路径，Full 适合作为精度优先路径。

定义：

```yaml
product_pose:
  realtime_model: auto
  analysis_model: full
```

## 6.1 自动选择规则

启动时进行 3 秒预热测试：

```text
Lite 推理 P50/P95
Full 推理 P50/P95
实际姿态 FPS
主线程长任务数
姿态检出率
```

建议规则：

```text
Full P95 <= 20 ms：
    使用 Full

Full P95 20～33 ms 且稳定：
    使用 Full 或允许用户选择

Full P95 > 33 ms：
    使用 Lite

设备发生持续过载：
    从 Full 降为 Lite，并明确提示
```

## 6.2 精度回归

对 8 个黄金视频分别比较：

* image landmarks 差异；
* 关节角差异；
* 动作候选数；
* 有效次数；
* `NO_REP`；
* `UNSURE`；
* 脚部事件；
* 触地结果；
* 3D Assist 可用率。

只有 Lite 不明显破坏关键动作规则时，才允许作为产品自动模式。

# 七、第五阶段：拆分显示滤波与分析滤波

> 实施状态（2026-07-23）：显示流与分析流已完全拆分。Python 正式分析流
> 使用独立 `responsive` One Euro 且强制 `prediction_enabled: false`；
> 浏览器显示流使用 `ultra_responsive`（`min_cutoff=2.2`、`beta=0.12`）、
> 速度/可见度相关 raw blend、节点分组和独立 image/world 状态，原始权重
> 上限为 `0.45`。动作、模型、身份或长断流会重置相应状态；显示 landmarks
> 不发送到 HYROX，也不写入正式报告姿态字段。

当前同一个 One Euro 输出不应该同时承担：

* 稳定动作判定；
* 极低延迟屏幕显示。

新增两个独立实例。

## 7.1 分析滤波器

```yaml
analysis_smoothing:
  profile: responsive
  prediction_enabled: false
```

用于：

* 状态机；
* 动作角度；
* 接触；
* 脚步；
* 计数；
* 报告。

保持现有行为，避免回归。

## 7.2 显示滤波器

新增：

```yaml
display_smoothing:
  profile: ultra_responsive
  raw_blend_enabled: true
  prediction_enabled: true
```

建议初始档：

```yaml
ultra_responsive:
  min_cutoff: 2.2
  beta: 0.12
  d_cutoff: 1.0
```

以上只作为实验起点，必须通过抖动—延迟曲线调参。

## 7.3 速度相关原始坐标混合

对高置信度关键点：

```text
速度低：
    更多使用 One Euro 输出

速度高：
    增加原始坐标权重
```

例如：

```python
speed_ratio = clamp(
    (joint_speed - slow_threshold)
    / (fast_threshold - slow_threshold),
    0.0,
    1.0,
)

raw_weight = (
    speed_ratio
    * visibility_weight
    * max_raw_weight
)

display_point = (
    filtered_point * (1.0 - raw_weight)
    + raw_point * raw_weight
)
```

建议：

```text
max_raw_weight = 0.45
```

节点分组：

```text
手腕、踝、脚跟、脚尖：
    允许较高 raw_weight

肩、髋：
    保持较低 raw_weight

鼻、耳、眼：
    不需要激进预测
```

# 八、第六阶段：显示层短时预测

> 实施状态（2026-07-23）：已完成浏览器本机姿态显示链路。独立
> `DisplayPosePredictor` 在视频帧回调中按姿态采集/呈现时间与
> `expectedDisplayTime` 动态预测 0～45 ms，并实现本节所需的置信度、
> 断流、人体尺度位移、反向衰减、核心/面部降权和支撑脚横向约束。
> 预测只传给 Canvas 绘制副本；发送 Python 的原始关键点和正式规则链
> 不变；分析协议还会拒绝 prediction/display landmarks 字段。自动化测试
> 与八视频黄金回归已通过；真实 sensor-to-photon
> 改善仍需按本方案用目标设备的 120/240 FPS 高速录像验收。

实现一个独立的：

```text
DisplayPosePredictor
```

只用于骨架显示。

## 8.1 预测目标时间

不要固定预测未来 50 ms。

根据真实结果年龄动态预测：

```text
prediction_horizon
= 当前预计显示时间
- 姿态对应帧的采集/显示时间
```

限制：

```text
0～50 ms
```

如果无法得到摄像头硬件采集时间，则使用：

```text
当前视频帧 presentationTime
- 姿态视频帧 presentationTime
```

## 8.2 速度模型

第一版采用常速度模型：

```text
predicted_position
= current_position
+ smoothed_velocity × prediction_horizon
```

速度不能直接用两帧原始差值，应先进行轻量平滑：

```text
velocity =
0.6 × previous_velocity
+ 0.4 × measured_velocity
```

## 8.3 预测限制

配置：

```yaml
display_prediction:
  enabled: true
  max_horizon_ms: 45
  max_body_scale_displacement: 0.06
  minimum_visibility: 0.70
  velocity_decay: 0.85
  disable_after_gap_ms: 100
```

规则：

* 低可见度节点不预测。
* 身份切换后立即重置。
* 超过 100 ms 没有新结果时不继续外推。
* 每个节点位移不能超过人体尺度的固定比例。
* 运动方向突然反转时降低预测强度。
* 关节接近运动极值时降低预测强度。
* 脚部处于稳定支撑状态时禁止横向漂移预测。
* 胸部、髋中心可以预测，但幅度小于四肢末端。
* 不预测可见度和置信度。
* 不把预测结果写入报告或 CSV 的正式姿态字段。

## 8.4 动作阶段感知预测

利用当前 HYROX 阶段进一步限制预测：

### Lunge

* 下降和上升过程中允许膝、髋短预测。
* 最低点附近降低预测，防止穿过地板或膝角过冲。
* 站立阶段主要抑制抖动。

### Wall Ball

* 双腕快速上升时允许较高预测。
* 最高点附近快速衰减预测。
* 脚底支撑点保持稳定。

### Burpee Broad Jump

* 腾空期间允许身体中心短预测。
* 落地前后迅速降低脚部预测。
* 接触规则仍只使用真实观测。

### Rowing / SkiErg

* 周期中段可以使用速度外推。
* Catch、Finish、顶部和底部端点附近减少预测。

* 协议拒绝显示预测字段进入状态机、计数、规则和正式报告。

## 神经预测研究：暂不实施

* 不建立训练数据采集或训练入口。
* 不实现 MLP、GRU、TCN、Transformer 或其他时序模型。
* 当前只保留显示层常速度 0–45 ms 短时预测。

每轮完成后执行：

```powershell
python -m src.doctor
python -m pytest -q
python tools/check_text_format.py
python -m src.smoke_test
python -m compileall -q hyrox src webui tools main.py
node --check webui\static\app.js
python -m build
pose-golden --report outputs\validation\hyrox_golden_report.json
```

最终报告必须包含：

1. 修改文件列表。
2. 优化前后的内部延迟。
3. 优化前后的高速录像延迟。
4. 视频帧和姿态帧的时间差。
5. Full 和 Lite 精度、速度比较。
6. 浏览器本地模式和服务器模式比较。
7. 显示预测的抖动和过冲结果。
8. 全部动作回归结果。
9. 487 项原测试及新增测试结果。
10. 未解决的摄像头和显示器硬件延迟。
