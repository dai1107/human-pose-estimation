## 4. 实现关节角计算

新增 `angles.py`。

实现通用三点夹角函数：

```python
calculate_joint_angle(point_a, point_b, point_c) -> float
```

其中角度为：

```text
A-B-C 中点 B 的夹角，范围 0 到 180 度。
```

至少计算：

```text
左肘角：左肩 - 左肘 - 左腕
右肘角：右肩 - 右肘 - 右腕

左膝角：左髋 - 左膝 - 左踝
右膝角：右髋 - 右膝 - 右踝

左髋角：左肩 - 左髋 - 左膝
右髋角：右肩 - 右髋 - 右膝

左肩角：左髋 - 左肩 - 左肘
右肩角：右髋 - 右肩 - 右肘
```

同时计算：

```text
骨盆中心
肩部中心
躯干向量：肩部中心 - 骨盆中心
```

基于躯干向量计算一个“躯干倾斜代理角”。该值应明确标注为视觉姿态代理指标，不能解释为医学测量值。

角度计算要求：

* 处理零长度向量；
* 对余弦值做 `clip(-1, 1)`；
* 不可计算时返回 `NaN`；
* 至少为 90°、180°、0° 的人工构造样本写单元测试。

## 5. 实现节段向量、速度与角速度

新增：

```text
segments.py
velocity.py
```

### 节段方向

至少计算：

```text
左/右大腿：髋 → 膝
左/右小腿：膝 → 踝
左/右上臂：肩 → 肘
左/右前臂：肘 → 腕
躯干：骨盆中心 → 肩部中心
```

每个节段应输出：

* 原始向量；
* 单位方向向量；
* 是否有效。

### 线速度

对于位置序列：

```text
v(t) = (p(t) - p(t-1)) / dt
```

至少计算：

```text
左/右手腕速度
左/右脚踝速度
骨盆中心速度
肩部中心速度
```

### 关节角速度

对于角度序列：

```text
omega(t) = (angle(t) - angle(t-1)) / dt
```

至少计算：

```text
左右肘角速度
左右膝角速度
左右髋角速度
```

要求：

* 使用真实时间戳计算 `dt`，不要假设固定帧率；
* 当 `dt <= 0`、时间间隔过大或关键点缺失时，跳过该帧导数；
* 导数计算必须基于平滑后的坐标或平滑后的角度；
* 不要把屏幕坐标像素速度错误解释为真实米每秒。


## 4. 深蹲状态机与重复计数

实现状态机：

```text
IDLE
→ READY
→ DESCENT
→ BOTTOM
→ ASCENT
→ COMPLETE
→ READY
```

状态定义：

```text
IDLE：
尚未完成校准或人体未稳定检测到。

READY：
人体稳定站立，等待开始动作。

DESCENT：
骨盆相对基线向下移动，且膝角或髋角出现持续屈曲趋势。

BOTTOM：
骨盆达到本次动作的最低位置附近，垂直速度接近零，且膝/髋屈曲达到局部最大。

ASCENT：
骨盆向上运动，膝角或髋角出现持续伸展趋势。

COMPLETE：
身体回到接近站立基线，完成一次重复。
```

要求：

* 不要仅靠单一膝角阈值计数；
* 使用骨盆垂直位移、膝角变化、髋角变化和时间连续性共同判断；
* 使用滞回机制，防止底部抖动被重复计算；
* 设置最短动作时长和最短阶段时长；
* 人体离开画面或关键点缺失时暂停状态机；
* 重新检测稳定后再恢复 READY，而不是继续旧动作。

状态机参数必须来自配置文件，例如：

```yaml
min_rep_duration_ms: 600
min_descent_duration_ms: 180
min_ascent_duration_ms: 180
bottom_hold_max_ms: 1200
baseline_return_tolerance: 0.15
```

这些参数仅用于动作分段稳定性，不作为“动作标准”定义。

## 3. 定义投篮动作阶段

实现状态机：

```text
IDLE
→ SETUP
→ DIP
→ RISE
→ ARM_EXTENSION
→ RELEASE_PROXY
→ FOLLOW_THROUGH
→ RECOVERY
```

阶段含义：

```text
IDLE：
人体未稳定检测到，或尚未处于分析片段。

SETUP：
身体稳定、持球准备阶段；不要求检测篮球。

DIP：
骨盆出现向下位移，膝角或髋角出现屈曲趋势。

RISE：
骨盆向上运动，髋膝踝开始伸展。

ARM_EXTENSION：
投篮侧上臂抬升，肘部出现持续伸展，手腕上升。

RELEASE_PROXY：
在未接入篮球检测的前提下，根据手腕速度峰值、肘部伸展、手腕高度和时间连续性估计的“出手代理时刻”。

FOLLOW_THROUGH：
出手代理后，手腕与肘部维持向上或向前延展，身体进入随挥阶段。

RECOVERY：
身体动作速度下降，回到稳定姿态。
```

要求：

* 不允许把 `RELEASE_PROXY` 命名为真实“球离手时刻”；
* 报告必须写明：这是基于人体关键点的估计时刻；
* 有篮球检测结果时，才可进一步标记 `BALL_RELEASE_CANDIDATE`；
* 阶段状态机必须允许某些阶段缺失，例如定点投篮可能 DIP 很小；
* 避免强制所有投篮都经过相同幅度的屈膝或起跳。

## 14. 单元测试与验收标准

至少新增测试：

```text
1. 模拟膝角屈伸、骨盆上下移动、肘角伸展和手腕速度峰值时，状态机能识别 DIP、RISE、ARM_EXTENSION、FOLLOW_THROUGH。
2. 手腕速度峰值和肘部伸展条件同时满足时，可以生成 release_proxy。
3. 低可见度或缺失关键点时，release_proxy 置信度降低或返回不可用。
4. 事件顺序比较能正确识别提前、滞后、缺失和重叠。
5. 多次模拟投篮可生成一致性统计。
6. 与个人参考投篮比较时可生成 DTW 对齐图和峰值时间差。
7. 侧面视角下，不输出高置信的横向手臂对齐结论。
8. 手动 `--release-ms` 必须优先于自动 release_proxy。
9. 任何分析失败都要保留数据质量与限制说明，不得产生伪精确结论。
```

最终验收：

```text
1. 用户可从会话中裁剪一次投篮。
2. 系统可识别主要动作阶段或给出阶段不确定提示。
3. 系统可输出下肢、躯干、肩、肘、腕的峰值时序。
4. 系统可输出出手代理时刻及其置信度。
5. 系统可比较多次投篮的一致性。
6. 系统可与个人参考投篮动作做时间对齐和差异分析。
7. 系统可导出结构化 JSON、CSV、曲线图、关键帧和 Markdown 报告。
8. 系统不声称测量真实力量、关节力矩、肌肉活动或绝对投篮标准。
```
