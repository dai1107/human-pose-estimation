# 模型优化方案：当前收敛版

当前项目不再扩展未验证的额外姿态模型。正式保留：

1. `mediapipe`
2. `yolo-pose`

另有实验性辅助链路 `yolo-roi-mediapipe`，它不是正式姿态 backend，不参与默认策略，只在复杂背景/目标锁定场景下单独测试。

核心原则：

- MediaPipe 仍是默认主模型。
- YOLO Pose 是动作级候选模型，默认使用 GPU。
- 不直接平均 33 点和 17 点模型结果。
- 新运动项目必须先用代表视频跑同一套 metrics，再决定默认 backend。
- 删除未验证、未完整落地的额外模型占位，降低维护成本和依赖风险。

## 一、backend 策略

支持：

```powershell
python main.py --backend auto
python main.py --backend mediapipe
python main.py --backend yolo-pose
```

普通实时主链路支持运行中切换：窗口打开后按 `B` 可在 `mediapipe` 和 `yolo-pose` 之间切换，摄像头保持打开。该功能仅用于 `--fusion none --person-detector none`，切换时会重置平滑器；公平对比仍应固定 backend 分别复跑同一段视频。

普通实时主链路默认启用稳定性保护：

```powershell
python main.py --pose-hold-frames 5 --occlusion-guard
```

- `--pose-hold-frames`：短暂识别不到 pose 时保留上一帧有效骨架，默认 `5`，设为 `0` 可关闭。
- `--occlusion-guard`：手腕/手指靠近身体关节并触发异常跳变时，短暂保留上一帧稳定身体点；可用 `--no-occlusion-guard` 关闭。
- 保护触发次数会写入 metrics：`stabilized_hold_count`、`occlusion_guard_count`。

`--backend auto` 的选择规则：

1. 如果指定 `--action-type`，优先按动作类型选择。
2. 如果读取 `--input-video`，可按视频文件名推断 HYROX 动作。
3. 不能识别时默认使用 `mediapipe`。

## 二、HYROX 当前默认策略

| 动作 | 推荐 backend | 原因 |
|---|---|---|
| 农夫行走 | `mediapipe` | 稳定性和速度综合更好 |
| 划船机 | `yolo-pose` | HYROX 对比中稳定性更好 |
| 投掷药球 | `mediapipe` | 33 点细节更适合动作分析 |
| 拉雪橇 | `yolo-pose` | HYROX 对比中稳定性更好 |
| 推雪橇 | `mediapipe` | 综合指标更好 |
| 波比跳远 | `mediapipe` | 动作幅度大但 MediaPipe 综合更稳 |
| 滑雪机 | `yolo-pose` | HYROX 对比中稳定性更好 |
| 负重箭步蹲 | `mediapipe` | 下肢细节和 33 点输出更适合 |

## 三、GPU 策略

YOLO Pose：

```powershell
python main.py --backend yolo-pose --yolo-device auto
```

当前 RTX 4060 环境中，`auto` 会解析为 GPU `0`。GPU/CPU 对比结果显示 YOLO Pose 使用 GPU 后端到端耗时有明显下降。

MediaPipe：

```powershell
python main.py --backend mediapipe
```

MediaPipe 当前继续走 CPU 路径。

实验性 YOLO ROI + MediaPipe：

```powershell
python main.py --backend mediapipe --person-detector yolo --fusion yolo-roi-mediapipe --detector-device auto
```

YOLO detector 可以用 GPU，但整体收益受 MediaPipe CPU 姿态推理限制。当前 HYROX 对比没有把它作为任何动作的默认推荐；它不参与 `--backend auto`，也不支持实时按 `B` 热切换。

## 四、对比评估指标

每次对比至少保存：

- `success_rate`
- `avg_keypoint_confidence`
- `missing_rate_shoulder`
- `missing_rate_hip`
- `missing_rate_knee`
- `missing_rate_ankle`
- `keypoint_jitter`
- `angle_jitter`
- `avg_inference_time_ms`
- `p95_inference_time_ms`
- `avg_end_to_end_latency_ms`
- `backend_device`
- `source_model_distribution`
- `stabilized_hold_count`
- `occlusion_guard_count`

优先级：

1. 成功率高。
2. 关键部位缺失率低。
3. `keypoint_jitter` 和 `angle_jitter` 低。
4. P95 推理耗时低。
5. 平均端到端延迟低。

## 六、后续新运动接入流程

1. 录制 3 到 5 段代表视频。
2. 分别跑 `mediapipe` 和 `yolo-pose`，只有多人、复杂背景或目标锁定需求明显时再单独测试 `yolo-roi-mediapipe`。
3. 查看 metrics 和带骨架回放。
4. 将该运动的推荐 backend 写入 `src/utils/backend_policy.py`。
5. 更新 README 和任务说明。

不要凭单次主观观感直接修改默认 backend。
