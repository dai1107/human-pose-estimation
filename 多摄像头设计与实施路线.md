# 多摄像头设计与实施路线

## 当前状态

日常实时分析仍以单摄像头为正式主流程。单摄像头已经可以通过 `--camera-view front|side`
选择评价标准，并可在窗口中按 `V` 切换。切换视角会重置动作状态；会话采集中禁止切换，
避免一份记录混用两套标准。

项目已经建立多摄像头基础层：`CameraSource` 描述摄像头编号、视角和镜像方式，
`MultiCameraPlan` 校验两个或多个来源，`MultiCameraCapture` 可以同时打开并读取所有摄像头，
同时报告帧间时间偏差。该层不混合二维坐标，也不在尚未标定时声称得到三维结论。

连接一台正面摄像头和一台侧面摄像头后，可先运行：

```powershell
python tools/check_multicamera.py --camera 0:front:mirror --camera 1:side:no-mirror --frames 60
```

这条命令只验收设备能否同时打开、持续读帧和满足时间容差，还不会启动双视角动作融合。

## 正式双视角运行的实施顺序

1. 每个摄像头独立采集线程、独立时间戳和断线重连；禁止共享非线程安全推理实例。
2. 每个视角独立姿态后端、平滑器和动作分析器，正面与侧面分别产出带置信度的结论。
3. 使用时间窗口对齐结果；超出容差时降级到单视角，不冻结整个实时界面。
4. 只在决策层融合：正面负责左右对称和膝/手同步，侧面负责深度、躯干角和前后位移。
5. 记录每条反馈来自哪个摄像头、哪个视角、哪个时间戳；冲突时优先显示更适合该指标的视角。
6. 完成双机位标定后，才能增加跨视角几何或三维指标，并必须用标注数据验证。

## 正式启用前的验收条件

- 两台摄像头连续运行 60 分钟，无持续内存增长或资源泄漏。
- 任意一台断开后，另一台继续工作并明确显示降级状态；恢复连接后可安全重新加入。
- 正面/侧面时间对齐成功率、最大偏差和丢帧率写入 metrics。
- 单摄像头全部回归保持通过，双摄像头功能不能成为默认启动依赖。
- 使用人工标注视频验证融合后确实优于最佳单视角，而不是简单增加提示数量。
