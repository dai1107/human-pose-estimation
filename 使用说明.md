# HYROX 使用说明

项目支持网页版和桌面版。普通使用、手机摄像头和临时分享优先选择网页版；本机调试、录制视频、批量回放与模型对比选择桌面版。

## 0. 安装

支持 Python `3.10–3.12`。核心、YOLO 和开发依赖分别锁定在
`requirements-core.txt`、`requirements-yolo.txt` 和 `requirements-dev.txt`：

```powershell
python -m pip install -r requirements-core.txt
python -m pip install -r requirements-yolo.txt
python -m pip install -e .
pose-doctor --json
```

安装后可使用 `pose-estimation`、`pose-web`、`pose-replay`、`pose-doctor`、
`pose-golden`、`pose-endurance`、`pose-camera-benchmark`、`pose-clean` 和 5 个 `pose-reference-*` 参考动作命令，也可以继续使用原有
`python main.py` 等仓库命令。默认模型、配置和网页资源包含在发布包中。

网页版会话、上传和报告默认写入当前目录的 `outputs/`，可用 `POSE_OUTPUT_DIR`
指定其他位置；桌面版使用 `--save-dir` / `--log-dir`，参考动作工具使用
`--output-dir` / `--root` 指定位置。`pose-doctor` 和 `pose-clean` 会跟随
`POSE_OUTPUT_DIR`。

## 今日更新（2026-07-23）

- 正式摄像头、上传视频和默认桌面链路已统一为 MediaPipe Pose；`auto` 只作为兼容值，YOLO/RTMW 不会在产品实时会话中自动启动。
- 8 个固定示例改为读取带文件指纹的 v2 姿态缓存，包含 image landmarks、可用的 world landmarks 和手部关键点；页面模型名为“预计算示例结果”，运行时推理为 `0.0 ms`。箭步蹲缓存中的 YOLO + MediaPipe 只描述离线生成及 image/world 身份匹配来源，不是播放时模型。
- 网页实时结果改为就绪即推送，并继续使用最新帧优先、单请求在途和过期结果拒绝；手指显示默认关闭。当前测试机测得摄像头协议往返延迟 P50 `18.7 ms`、P95 `35.5 ms`，原轮询链路约 `58 ms`。
- 正式分析和网页显示现在使用两套独立滤波：分析流保持 `responsive` 且禁止预测；显示流使用 `ultra_responsive` 和速度相关原始坐标混合。网页关节旁只显示通过质量门的三维角度并标注 `3D`；MediaPipe world landmarks 同时进入 3D Assist，但不会替换二维动作阶段或阈值。
- 示例和上传视频继续逐帧分析，并按源视频 FPS 原速展示；预计算缓存只降低运行时推理成本，不会再造成倍速播放。
- 桌面实时入口完成模块化，新增 `pose-golden` 八视频黄金回归和 `pose-endurance` 30/60 分钟耐久验收入口。

上述延迟是旧服务器兼容姿态在 640 px、JPEG 0.65、本机 30 帧协议探针中的结果，不代表新的浏览器本机姿态链路。新链路应使用完整延迟审计重新建立设备基线。示例视频虽然不再推理模型，仍要完成解码、规则分析、骨架绘制和画面编码，因此不会瞬间结束。

### 前九阶段实施状态

1. **完整延迟审计**：网页和桌面均保存完整时间线、姿态年龄、视频年龄及两者差值；真实硬件总延迟仍需手机 120/240 FPS 高速录像。
2. **视频帧驱动显示**：支持的网页使用 `requestVideoFrameCallback` 同时驱动采集和骨架绘制，帧号、显示时间和预计显示时间全程关联；不支持时明确回退 `requestAnimationFrame`，迟到结果不会覆盖新帧。
3. **浏览器 MediaPipe Worker**：本机优先运行 Pose Landmarker，Worker 忙时只保留最新帧，画面立即使用本机结果，Python 只接收原始 image/world landmarks 做规则；服务器回退受 `server_pose_fallback` 控制。
4. **Lite/Full 产品档**：自动档执行约 3 秒设备基准，但 Lite 的原黄金区间只有 `3/8` 通过、严格等价门只有 `1/8`，所以正式自动档仍为 Full；Lite 只供显式速度实验。
5. **显示/分析双流**：分析流使用独立 `responsive` One Euro、无预测；显示流使用 `ultra_responsive`（2.2/0.12）并按速度、可见度及节点组最多混入 0.45 原始坐标。显示结果不进入计数、触地、状态机或正式报告姿态字段。
6. **显示层短时预测**：独立 `DisplayPosePredictor` 按姿态采集/呈现时间到浏览器预计显示时间动态预测 0–45 ms；低可见度、100 ms 断流、身份切换时禁用或重置，并限制人体尺度位移、反向过冲和支撑脚横向漂移。预测只传 Canvas，协议拒绝 prediction/display landmarks。
7. **主线程渲染优化**：骨架保持视频帧级更新；指标降至 5 FPS、统计 3 FPS、3D 角度文字 12 FPS，反馈仅内容变化时重建。绘制使用固定关键点缓冲区并缓存连接、字体和画布变换，同时记录渲染、Canvas、DOM P95 及 Long Task 阶段；实际数值需在目标设备采样。
8. **摄像头配置与硬件诊断**：网页优先请求 640×480@60、最低 30 FPS，失败时明确回退 30 FPS；页面保存实际摄像头设置并诊断实际 FPS、帧间隔、低光和重复帧。桌面提供 default/DSHOW/MSMF 与 MJPG/YUY2 基准工具和设备级缓存。
9. **本地优先架构收口**：网页本机 Worker 负责姿态与即时绘制，Python 只接收原始关键点执行 HYROX；桌面链路全部本机完成。服务器姿态回退由配置控制，预测协议字段被拒绝。

分析流的 `prediction_enabled` 仍强制为 `false`；显示预测只生成 Canvas 绘制副本，发送 Python 的原始关键点、动作状态机、计数、触地和正式报告均不使用预测姿态。神经网络、时序模型与训练流程均未实现，`local_first.neural_prediction_enabled` 只能为 `false`。
阶段收口后的自动化基线为 Python `530 passed`、Node `16 passed`，Full 黄金视频回归为 `8/8`；这些结果不替代目标设备摄像头和显示器的硬件延迟实测。

## 1. 网页版

本机启动可双击 `启动网页.bat`，或在命令行运行：

```powershell
.venv\Scripts\python.exe start_web.py
```

浏览器会打开 `http://127.0.0.1:5000`。在页面中选择画面来源、动作、视角和灵敏度，确认模型状态后再开启摄像头或选择视频。

产品网页只需要 MediaPipe 核心依赖。YOLO 与 RTMW 仅供离线比较、缓存重新生成和研究实验；需要这些实验后端时再安装对应依赖：

```powershell
# 实验性 YOLO 后端
.venv\Scripts\python.exe -m pip install -r requirements-yolo.txt

# 实验性 RTMW：CPU 或 NVIDIA GPU 二选一
.venv\Scripts\python.exe -m pip install -r requirements-rtmw-cpu.txt

.venv\Scripts\python.exe -m pip uninstall -y onnxruntime onnxruntime-gpu
.venv\Scripts\python.exe -m pip install -r requirements-rtmw-gpu.txt
```

RTMW 权重文件名、官方地址和 GPU 检查方式见 [models/README.md](models/README.md)。安装这些实验依赖不会让产品网页出现模型切换项。

### 实时语音改进提示

- “动作反馈”右上角提供“语音开/关”，默认开启，并记住当前浏览器的选择；
- 每完成人体规则候选或分析周期时，系统会汇总该段动作最需要改进的内容并播报，单次最多两条；
- 建议与文字报告中的“逐次动作表现”使用同一套汇总规则，包括持续出现的动作问题和清晰关键阶段的角度偏离；
- 本次没有持续性问题时不播放语音，避免训练过程中产生无效干扰；
- Farmers Carry 不按次数拆分，问题持续约 1.2 秒后即时提示，同一问题设有 8 秒重复冷却；
- 语音由浏览器在本机合成，不申请麦克风、不上传音频，也不会写入本机录制的视频。

推荐使用较新的 Chrome、Edge 或 Safari。若页面显示“语音不可用”，说明当前浏览器没有提供 Web Speech API，可更换浏览器或继续只看文字反馈。

展开“高级设置”后，可以分别调整：

- **骨架设置**：选择完整骨架、仅上半身或仅下半身；
- **识别模型**：摄像头和上传视频固定使用 `MediaPipe Pose`；固定示例显示“预计算示例结果”，不在播放时加载模型；
- **显示手指节点**：需要时启用；摄像头和上传视频使用独立 MediaPipe 手部模型，固定示例读取手部缓存。每只手显示五根手指的 20 个非手腕关节点，默认关闭以降低实时延迟；
- **隐藏面部**：隐藏面部姿态节点，默认关闭；
- **镜像画面**：控制画面左右翻转，摄像头模式默认开启，示例和上传视频默认关闭。

“显示手指节点”“隐藏面部”和“镜像画面”是同级独立开关，不再放在“骨架设置”下拉项中。选择仅上半身或仅下半身时，未包含在该骨架范围内的节点仍不会显示。

摄像头和上传视频的正式识别模型只有 `MediaPipe Pose`，旧版 `auto` 请求会直接映射到 MediaPipe。固定示例使用带视频、模型和 schema 校验的 v2 预计算关键点缓存，包含可用的 MediaPipe world landmarks；其中箭步蹲缓存离线使用 YOLO 锁定前景人物，并匹配同一人的 MediaPipe image/world 关键点。该组合模型不会在示例播放时或实时产品会话中启动。

实时链路默认请求 640×480@60 摄像头并声明最低 30 FPS；约束不满足时会明确提示并回退理想 30 FPS。页面记录实际轨道设置、呈现 FPS、帧间隔、低光和重复帧。浏览器本机 MediaPipe 最多运行 30 FPS；Worker 忙时只保留一个最新待处理帧，本机关键点立即绘制，只把原始 image/world landmarks 和时间戳发送给 Python 做动作规则，不会再为同一帧在服务器重复运行 Pose Landmarker。“本机姿态档位”默认使用约 3 秒自动基准；基准期间只绘制骨架，不进入动作状态机。当前 Lite 只通过 `1/8` 严格黄金档位门，因此自动模式保持 Full，Lite 仅供用户明确选择速度优先实验。如果 Worker、WASM 或模型初始化失败、连续推理异常或本机推理持续过慢，且配置允许 `server_pose_fallback`，页面会明确显示“服务器兼容姿态”，再使用原 640 px、JPEG 0.65、单请求在途链路；配置禁止回退时直接显示错误。需要最低延迟时保持“显示手指节点”关闭。

临时公网使用双击 `启动公网访问.bat`；如需随机访问口令：

```powershell
.venv\Scripts\python.exe start_public_web.py --protected
```

公网脚本生成的 Quick Tunnel 地址每次启动都会变化。局域网中的普通 HTTP 地址通常不能让其他电脑或手机获得浏览器摄像头权限，跨设备使用应通过 HTTPS 公网地址。摄像头权限、隐私、报告下载、故障排查和部署说明见 [网页版使用说明.md](网页版使用说明.md)。

网页版可下载文字报告、JSON 和 CSV。文字报告包含每次动作做得好的地方与需要改进的地方；语音只播放改进建议，不朗读整份报告。

## 2. 桌面版启动

激活环境：

```powershell
.\.venv\Scripts\Activate.ps1
```

使用摄像头：

```powershell
python main.py --hyrox-action wall_ball --camera-view front
```

桌面摄像头后端默认使用 `--camera-api auto`：精确匹配设备缓存，缓存缺失时从 OpenCV 默认后端开始，不再写死 DSHOW。需要在当前设备建立基准时，由用户主动运行：

```powershell
pose-camera-benchmark --camera 0
```

该命令会实际打开摄像头，并测试 default/DSHOW/MSMF、MJPG/YUY2 和三种分辨率/帧率组合。输出的 `sensor_to_photon_ms` 默认保持 `null`；仍需 120/240 FPS 外部录像才能填写物理总延迟。

使用视频：

```powershell
python main.py `
  --input-video "HYROX视频\药球.mp4" `
  --hyrox-action wall_ball `
  --camera-view front `
  --no-mirror
```

## 4. 拍摄建议

- 一次只保留一名主要运动者；
- 身体尽量完整入镜，避免脚、手和头部长期出画；
- `front` 适合观察左右对称、膝部轨迹和站姿；
- `side` 适合观察躯干角度、髋铰链、步幅和前后阶段；
- 斜视角使用 `front_left` 或 `front_right`；
- 视频输入通常使用 `--no-mirror`，摄像头预览可保留默认镜像。

视角变化会重置当前 HYROX 分析器，避免混用不同视角的阈值。视角不足以支持某项判断时会显示 `CAMERA_VIEW_LIMITED`。

## 5. 灵敏度与配置

```powershell
python main.py `
  --hyrox-action lunge `
  --hyrox-sensitivity high `
  --camera-view side
```

- `low`：更保守，减少误计数；
- `medium`：默认；
- `high`：更敏感，适合动作幅度较小或较快的视频。

自定义动作配置：

```powershell
python main.py `
  --hyrox-action rowing `
  --hyrox-config configs\hyrox\rowing.yaml `
  --camera-view side
```

## 7. 各动作指导范围

- **Lunge**：验证后膝触地、触地后双膝/双髋伸展、有效触地腿交替及无额外调整步。
- **Wall Ball**：验证站直起始、髋低于膝、向上完全伸展和双手投掷代理；不检测药球、命中或目标高度。
- **Rowing**：记录划船分析周期并提供腿—躯干—手臂时序反馈；训练检测区间内可提示持续站起代理，不读取功率、距离或阻力。
- **SkiErg**：记录拉动分析周期和髋铰链反馈；不识别底座，因此不因脚离地判比赛违规。
- **Burpee Broad Jump**：验证胸部触地代理、双脚同步起落、起落错位、手部位置、无补步/碎步及前向跳跃。
- **Sled Push**：记录 `drive/step` 分析周期和技术反馈；不以固定关节角度或躯干姿势判比赛违规。
- **Sled Pull**：记录拉动分析周期，并检测拉动阶段持续跪姿、明确坐姿和证据不足的可能坐姿。
- **Farmers Carry**：连续监控搬运状态、躯干稳定和手臂是否伸展并位于身体两侧；不检测壶铃重量或真实距离。

### 程序如何定位动作问题

用户必须先选择动作；程序不会自动识别当前属于哪项 HYROX 动作。选择后，摄像头或视频的每一帧会按以下顺序处理：

1. 姿态模型输出原始人体关键点和置信度；原始点分别进入正式分析滤波和低延迟显示滤波。
2. 系统根据关键点计算关节角度、躯干方向、左右高差、身体中心、归一化距离和可见度；需要时再计算局部地板、膝/胸触地代理和左右脚事件。
3. 动作专属状态机根据连续时序确定当前阶段。过渡阶段可以容忍短时漏检，但最低点、伸展点、起跳或落地等关键端点必须实际观察到；实时“中/高”灵敏度允许一个已处理帧确认短暂端点，“低”灵敏度更保守。
4. 网页当前阶段参考标准用于骨架颜色和角度标记；动作分析器使用动作专属阈值另行生成实时技术反馈。不适用于该阶段的标准不会参与当前姿态评价。
5. 完成一个候选后，计数动作再聚合必需人体规则，得到 `VALID`、`NO_REP` 或 `UNSURE`。
6. 最后检查可见度、关键点置信度、地板、视角和证据持续时间；不可靠结论统一降级为 `UNSURE`。

页面或桌面叠加层中的结果应这样理解：

| 显示内容 | 表示什么 | 是否直接影响计数 |
|---|---|---|
| 骨架颜色和关节角度 | 当前阶段是否明显偏离参考范围或命中高置信度问题 | 不直接决定；边界值和无法评价状态会保持中性 |
| “动作反馈”文字 | 当前帧或最近动作段命中的优先技术问题 | 不一定；多数属于训练建议 |
| 有效/未完成/无法确认 | 完整候选的必需人体规则和证据质量 | 是，用于 `VALID/NO_REP/UNSURE` |
| “动作标准” | 当前实现采用的官方人体代理与训练参考范围 | 用于解释，不代表器械、距离或现场条件已通过 |

例如 Lunge 的“躯干前倾过多”可以作为技术反馈出现，但它本身不是五项计数必需规则之一；后膝未触地、触地后未伸展、没有交替或出现额外调整步才会影响完整候选。完整原理与中灵敏度示例见 [README 的动作问题判断流程](README.md#动作问题如何判断)，所有默认阈值见 [动作配置说明](configs/hyrox/README.md)。

### 当前程序的计数与分析周期标准

以下是默认“中”灵敏度下的二维人体代理。Lunge、Wall Ball 和 Burpee Broad Jump 形成
人体规则候选；Rowing、SkiErg、Sled Push 和 Sled Pull 只形成分析周期；Farmers Carry
不按次数拆分。器械重量、目标命中、真实距离和完成线仍需人工或器械确认。

| 动作 | 必须出现的关键端点 | 当前判定 |
|---|---|---|
| 负重箭步蹲 | 站立 → 后膝触地确认 → 触地后完全伸展 | 正面证据要求双侧约 ≥165°；推荐侧面视角使用更可见的同侧腿链和 3° 二维容差，并连续保持。触地腿还需交替且不得出现额外调整步。 |
| Wall Ball | 站直起始 → 髋低于膝 → 向上伸展 → 双手投掷代理 | 起始双膝/双髋约 ≥165°且躯干接近竖直；最低点基于局部地板确认髋低于膝；投掷端点双膝/双髋约 ≥165°，双腕从胸前同步上升并均高于肩。 |
| Rowing | Catch → Finish | Finish 时增加 `cycle_count`；Drive 可作为过渡。它是分析周期，不是官方有效次数。 |
| SkiErg | 顶部 → 底部 → 顶部 | 返回顶部时增加 `cycle_count`。它是分析周期，不是官方有效次数。 |
| Burpee Broad Jump | 胸部触地确认 → 双脚同步起跳 → 双脚同步落地 → 下一次 hands-down 验证 | 落地后先进入 `AWAITING_NEXT_HANDS`，继续检查补步/碎步；还要求合法手部位置代理、起落错位代理，以及身体中心和双脚发生同向前移。 |
| Sled Push | Drive → Step | 每个推动步增加 `cycle_count`；它不是 50 m 完成数或官方有效次数。 |
| Sled Pull | Reach → Pull → Recover → Reach | 后拉并向前回正、再次到达前伸位后增加 `cycle_count`；它不是官方有效次数。 |
| Farmers Carry | 连续搬运状态 | `cycle_count` 和 `rep_count` 保持 0；查看 `carrying/rest`、反馈和持续违规状态。 |

### 结果字段与“不确定”

- `candidate_count`：已完成的人体规则候选或兼容分析候选；
- `pose_valid_rep_count`：二维人体规则通过数；不代表器械、目标、距离等官方条件也通过；
- `no_rep_count`：证据充分且必需人体规则明确失败；
- `unsure_count`：遮挡、地板失效、视角不适合、置信度不足、单帧异常或关键端点丢失，无法可靠下结论；
- `rep_count`：兼容字段。距离动作还必须查看 `cycle_count`、`count_semantics` 和 `official_rep_count_supported`，不能把界面的“完整动作周期”解释成官方有效次数。

对人体规则计数动作，三类结果互斥并满足：

```text
candidate_count
= pose_valid_rep_count + no_rep_count + unsure_count
```

技术质量提示和必需人体规则是两层概念。轻微膝内扣、节奏或对称性提示不会自动取消计数；
但后膝未触地、Wall Ball 必需深度/伸展失败、Burpee 缺少前向跳跃等必需规则会决定最终
`NO_REP` 或 `UNSURE`。

### 距离动作的人体违规代理

- Rowing：`ROWING_EARLY_STAND_PROXY` 只覆盖用户点击开始到停止分析的训练区间，持续至少约 300 ms 才激活；
- Sled Pull：`SLED_PULL_KNEELING_VIOLATION` 和 `SLED_PULL_SEATED_VIOLATION` 只在证据持续且明确时激活；坐姿几何成立但地板/膝接触不清晰时输出 `UNSURE_POSSIBLE_SEATED_PULL`；
- Farmers Carry：移动期间手臂未基本伸展或未位于身体两侧持续约 300 ms，分别输出 `ARM_NOT_EXTENDED_VIOLATION`、`ARM_NOT_BY_SIDE_VIOLATION`；
- SkiErg 和 Sled Push 不新增不可靠的人体比赛违规判断。

### 实时摄像头计数建议

- 修改代码或配置后必须重启网页服务或桌面程序，旧进程不会自动加载新逻辑。
- 实时摄像头的“中/高”灵敏度使用一个已处理帧确认关键端点，适合投球、起跳等短暂姿态；“低”灵敏度仍要求连续多帧，漏计时先切换到“中”。
- Wall Ball 开始前先站立，让系统建立起点；使用正面或斜前方视角，确保脚踝、髋、肩和头顶投球时的手腕同时入镜。
- Farmers Carry 是连续项目；次数一直为 0 属于设计行为，应查看“搬运中/休息”和姿态反馈。
- 完整技术阈值和自定义字段见 [configs/hyrox/README.md](configs/hyrox/README.md#默认中灵敏度计数与分析周期端点)。

## 9. 常见问题

没有识别到动作：

- 确认动作与视频内容一致；
- 确认人物完整入镜且没有明显多人干扰；
- 检查拍摄视角是否设置正确；
- 尝试提高灵敏度；
- 桌面版可使用 `--hyrox-debug` 查看姿态特征。

结果经常是 `UNSURE`：

- 确保决定性部位持续入镜：Lunge 的膝和脚、Burpee 的胸部/双手/双脚、Wall Ball 的双腕和脚下地板；
- 选择正确拍摄视角，并在相机移动后重新建立或手动标定地板线；
- 不要把 `UNSURE` 当作 `NO_REP`，它表示当前证据不足。

Rowing、SkiErg 或雪橇“次数”与预期不同：

- 这些项目显示的是 `cycle_count` 分析周期，不是官方距离、雪橇过线或有效次数；
- Farmers Carry 始终不按次数拆分，`rep_count` 为 0 是设计行为。

计数过多：

- 改用 `low` 灵敏度；
- 固定相机，减少抖动；
- 避免中途切换视角或镜像。

检查环境与完整参数：

```powershell
python -m src.doctor
python main.py --help
python -m pytest -q
```

配置文件在启动时严格校验。字段拼错、类型错误、数值越界、重复字段、动作名与文件
不一致都会显示 `CFG001` 并以退出码 `2` 结束；先运行 `python -m src.doctor` 可一次
检查全部 HYROX 和参考动作 YAML。自定义动作配置可以省略未修改字段，但不能加入未知字段。

桌面版、网页版和回放工具的滚动日志默认位于 `outputs/logs/`，每个日志最多 2 MiB，
保留 5 个备份。普通模式只显示可执行的错误提示；需要完整 traceback 时使用：

### 黄金视频与耐久验收

仓库内 8 段 HYROX 视频都有固定回归区间，配置位于
`configs/hyrox_golden_videos.json`。修改动作状态机、平滑、模型或特征提取后运行：

```powershell
pose-golden --report outputs\validation\hyrox_golden_report.json
```

任一视频的帧数、姿态检出率或动作候选/周期计数超出区间时命令返回非零退出码；报告会
列出实际值、期望区间和失败字段。若明确变更了算法语义，应先人工复核视频，再有依据地
更新区间，不能为通过测试而直接放宽。

性能与资源耐久使用同一段本地视频循环，不依赖摄像头：

```powershell
# 仅确认工具链可运行，不代表通过正式耐久验收
pose-endurance --duration-seconds 10 --report outputs\validation\endurance_smoke.json

# 正式验收
pose-endurance --minutes 30 --report outputs\validation\endurance_30m.json
pose-endurance --minutes 60 --report outputs\validation\endurance_60m.json
```

报告记录 FPS、P95 帧延迟、内存起止值/峰值/增长、异常读帧率和输出完整性。不同硬件可
用 `--min-fps`、`--max-p95-latency-ms`、`--max-memory-growth-mb` 与
`--max-read-failure-rate` 设置经实测确认的验收阈值。正式结果必须保留 30/60 分钟
两份报告；秒级结果只能作为提交前冒烟。

退出码约定为：`0` 正常、`1` 未预期运行错误、`2` 参数/配置错误、`3` 摄像头或视频
输入错误、`4` 后端初始化错误、`5` 输出写入错误、`130` 用户中断。错误编号按
`CFG`、`SRC`、`BCK`、`OUT`、`RUN` 分类，可直接在日志中搜索。

活动会话遇到中断仍会尝试保存。会话 `metadata.json` 的 `write_status` 为
`complete` 才表示全部输出完成；`partial` 表示只保留了可恢复文件，此时查看
`recovery_error` 和 `recovered_files`，并先解决磁盘空间或权限问题。

所有结构化 JSON/CSV 都带 `schema_version` 和 `program_version`，JSON 还带
`artifact_type`。历史无版本文件按 schema `0` 读取；若文件版本高于当前程序，会显示
`SCH001`，此时应升级程序，不要手工猜测字段。

清理输出前先预览：

```powershell
pose-clean --json
pose-clean --older-than-days 30 --max-total-gb 20
pose-clean --older-than-days 30 --max-total-gb 20 --apply
```

不加 `--apply` 永远不会删除。默认保留策略按类别为 2–90 天；清理范围只包括已登记的
生成目录，不处理模型、配置、输入视频或其他未知目录。

## 10. 模型、平滑与参考比较

- **MediaPipe Pose**：默认姿态识别后端，模型文件为 `models/pose_landmarker_full.task`；
- **YOLO11n Pose**：实验性离线比较后端，模型文件为 `yolo11n-pose.pt`；
- **YOLO + MediaPipe / YOLO + RTMW WholeBody**：保留为实验、benchmark 和研究消融实现，不属于产品网页或默认桌面链路；
- **手机端运行方式**：支持的浏览器在本机 Worker 中运行 MediaPipe，只发送关键点给服务器做 HYROX 规则；设备不支持或模型加载失败时明确回退服务器兼容姿态。手机摄像头、WASM 性能、发热和网络仍会影响体验；
- **分析 One Euro**：使用姿态对应帧的真实采集时间差，提供 `stable`、`balanced`、`responsive` 三档，正式规则默认 `responsive`；image/world 状态独立，观测间隔超过 250 ms 时重置，预测强制关闭；
- **显示 One Euro 与 raw blend**：网页显示固定从 `ultra_responsive`（`min_cutoff=2.2`、`beta=0.12`）起步。速度越快、可见度越高，越多混入原始坐标，最大 0.45；腕踝、脚跟和脚尖使用完整权重，肩髋为 0.35 倍，面部为 0。该结果只绘制，不进入 Python 动作分析；
- **3D Assist Mode**：MediaPipe world landmarks 用于并行计算膝、髋、肘、肩三维角度；网页角度叠加只显示通过质量门的三维关节角并标注 `3D`，缺失或不可靠时隐藏，不以二维数值代替。可靠且与二维一致时只增强相关角度规则的置信度，严重冲突时把相关候选降为 `UNSURE`，缺失或不可靠时动作判定完整回退二维。动作阶段、二维角度阈值和逐规则 PASS/FAIL 不会被 3D 替换，触地、地板、起落、腕部时序、补步和距离规则仍只用二维；桌面 `kinematics_3d.csv`、`summary.json` 与网页 JSON/CSV 报告会保存 Assist 状态、2D/3D 差值、冲突比例和失败原因；
- **DTW**：参考动作比较中的时间序列对齐方法；
- **CAMERA_VIEW_LIMITED**：当前视角不足以可靠判断某项动作标准时的提示码。

多摄像头使用前可运行：

```powershell
python tools/check_multicamera.py --camera 0:front --camera 1:side
```

其中 `check_multicamera.py` 用于检查多路画面的时间偏差，不替代动作分析。

### 个人参考动作命令

桌面版按 `C` 记录的会话默认位于 `outputs/sessions/SESSION_ID/`。先检查会话质量，
再从动作时间段创建个人参考：

```powershell
pose-reference-inspect --session outputs\sessions\SESSION_ID
pose-reference-create --session outputs\sessions\SESSION_ID --start-ms 1000 --end-ms 5000 --name "我的标准动作" --action-type lunge --camera-view side
pose-reference-list
```

比较候选会话并导出参考动作：

```powershell
pose-reference-compare --session outputs\sessions\CANDIDATE_ID --reference outputs\references\REFERENCE_ID --start-ms 1000 --end-ms 5000
pose-reference-export --reference outputs\references\REFERENCE_ID
```

参考动作默认写入 `outputs/references/`，比较报告默认写入 `outputs/comparisons/`；
可用 `--output-dir` 或 `--root` 改变位置。比较过程会按人体相对尺度归一化并使用
DTW 对齐；输入质量不足时应先处理 `pose-reference-inspect` 给出的警告。
